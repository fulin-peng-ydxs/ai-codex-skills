# Page UX Audit

审查页面体验与视觉交付质量。聚焦“稳定、统一、直观、克制、细节完整”，不把业务规则正确性当作审查目标。

## 执行流程

1. 确定审查范围：
   - 用户指定模块、路由、页面文件、PR 或变更范围时，只审查指定范围。
   - 用户未指定范围时，默认审查当前项目所有可访问页面；从路由表、菜单配置、页面目录、页面注册表和项目文档中识别页面。
   - 优先覆盖新增页面、近期改动页面、用户可导航页面和核心工作流页面。
2. 读取项目页面规范：
   - 优先读取 `DESIGN.md`、`AGENTS.md`、`README.md`、页面模板、布局规范、组件目录、主题 token、同类页面实现。
   - 若项目没有明确设计规范，以当前项目中质量最高、出现频率最高的同类页面作为一致性基准。
3. 判断视口范围：
   - 项目声明桌面后台、管理系统或数据工作台，且没有移动端目标时，不额外审查移动端。
   - 项目声明响应式、移动端、H5、小程序，或用户明确要求移动端时，必须检查移动端视口。
   - 无法确认时，在报告中说明未发现移动端目标，因此未按移动端阻塞项审查。
4. 尽量实测页面：
   - 使用浏览器、Playwright、截图、DOM 检查或项目已有预览方式打开页面。
   - 不只看代码下结论；无法实测时说明原因，并把结论降级为静态审查。
5. 按完整清单审查：
   - 读取 `references/audit-checklist.md`。
   - 覆盖风格统一、排版、颜色、字体、交互、状态、动画、说明、控件必要性和冗余交互。
6. 输出可执行报告：
   - 每个问题写清位置、现象、原因、建议和优先级。
   - 不使用“更现代”“更高级”“更有设计感”这类空泛评价。
   - 用户只要求审查时不要改代码；用户要求修复时再实施改动并复测。

## 核心原则

- 把风格统一当作硬约束。已有组件、token、布局、空状态、异常态、表格模式、弹窗模式、抽屉模式、按钮风格、间距尺度或状态样式能覆盖时，要求复用。
- 最小化交互承载。一个用户意图只保留一个主要交互路径；文字、状态、行点击、标签点击、行内操作、键盘操作或附近已有控件能自然承载时，不额外加按钮。
- 优先语义直达。代表可导航、可筛选、可操作含义的状态文字，在安全且可发现时可以直接作为交互目标；除确认、权限、破坏性或异步命令外，不再旁边重复放“查看 / 搜索 / 打开”按钮。
- 拒绝装饰性功能。控件、卡片、统计块、开关、筛选、动画或说明不支持明确用户任务时，应建议移除、合并、折叠或降级。
- 高密工具保持工作导向。后台、仪表盘、数据、管理和研究类页面优先保证可扫描、布局稳定、样式克制、控件可预期，而不是营销式构图。
- 检查状态而不是只看正常态。加载、空数据、错误、禁用、选中、hover/focus、提交中、成功、长内容、极端值、无权限或不可用状态都属于页面交付范围。

## 优先级

- `P0`：页面不可用、关键内容遮挡、主流程无法完成、严重溢出、白屏、交互阻断。
- `P1`：主操作不清楚、关键状态缺失、严重风格不统一、冗余控件造成理解混乱、用户容易误操作。
- `P2`：排版、间距、颜色、文案、动画、空态、表格、表单或弹层细节不专业，但不阻断使用。
- `P3`：可选打磨或长期一致性优化。

## 边界

- 项目没有移动端目标时，不要求移动端支持。
- 不审查业务算法、数据正确性、领域规则、权限策略或后端行为；除非它们直接影响页面可用性。
- 不用新增复杂功能来修复简单体验问题。
- 不基于个人审美要求重做页面；所有问题必须落在一致性、可用性、清晰度、稳定性或不必要复杂度上。
