# 页面 UX 审查清单

审查时使用本清单。只报告影响可用性、一致性、清晰度、稳定性或造成不必要复杂度的问题。

## 1. 范围与证据

- 从路由、菜单、页面注册表、文件名、组件预览、文档中识别页面。
- 用户指定模块、路由、页面文件、PR 或 diff 时，严格按指定范围审查。
- 用户未指定范围时，默认审查当前项目所有可访问页面。
- 先读设计事实再评价页面：设计系统、布局模板、主题变量、组件目录、全局样式、高质量同类页面。
- 优先使用浏览器证据：截图、视口检查、真实点击、hover/focus 检查、状态切换。
- 无法运行浏览器时，把结果标注为静态审查，并按证据强度控制结论。

## 2. 风格统一

风格统一是硬约束。项目已经有模式时，除非当前页面有充分理由，否则必须复用。

检查：

- 页面标题区、工具栏、筛选区、内容区、分页、空状态、弹窗、抽屉、标签页、表格、卡片、表单是否和同类页面一致。
- 同一含义的按钮是否使用一致的颜色、尺寸、图标风格、顺序、禁用态、加载态和危险态。
- 表格是否使用一致的行高、表头样式、排序提示、对齐、空态、加载态、分页和操作列行为。
- 表单是否使用一致的 label 位置、宽度、间距、必填标记、placeholder、校验时机和错误位置。
- 状态标签、徽章、状态点、chip、风险标签、结果文字是否共享语义颜色和形态规则。
- 弹窗和抽屉是否共享宽度、标题层级、内容 padding、footer 布局、滚动行为和关闭行为。
- 空状态是否复用项目级空态组件或类名；已有全局模式时，不临时自绘页面私有空态。
- 页面私有 CSS 是否重复实现全局工具类、设计 token、组件样式或既有布局模式。
- 是否写死颜色、间距、高度、阴影、渐变、圆角，绕开 token 或共享样式。

定级：

- 页面像另一个产品、或重复造已有组件行为：`P1`。
- 局部细节不一致但页面仍可理解：`P2`。
- 长期精致度问题：`P3`。

## 3. 布局与信息层级

检查：

- 首屏是否展示真实工作区或核心内容，而不是装饰、空白或大段说明。
- 用户是否不用读长文就能识别页面目的、主要内容、主操作和次操作。
- 相关控件和内容是否自然分组。
- 列、卡片、表单行、按钮、表头、操作区是否对齐稳定。
- 间距是否来自小规模尺度，不随机忽大忽小。
- 页面密度是否符合产品类型；管理、数据、研究、仪表盘、运营工具应紧凑、安静、可扫描。
- 滚动归属是否清楚，避免多个嵌套滚动区互相抢滚动。
- 固定区和滚动区是否有意设计；关键筛选、表头、分页、操作 footer 不应意外消失。
- 卡片是否只用于独立内容块，而不是装饰每一个页面 section。
- 避免卡片套卡片、边框过多、阴影过重、大面积无效空白和装饰容器。
- 加载、hover、切换 tab、筛选变化、刷新数据时，容器尺寸是否稳定。

## 4. 颜色与视觉语义

检查：

- 主色、次色、成功、警告、危险、禁用、选中、hover、focus 是否有稳定含义。
- 颜色是否遵循项目 token 或主题变量。
- 文字在常规、hover、选中、禁用、深浅主题下是否有足够对比度。
- 危险操作是否明确，但不在非当前任务中抢占注意力。
- 同一状态是否在不同页面使用同一颜色。
- 图表、徽章、标签、状态点是否必要时配合文字或形状，不只靠颜色传递关键信息。
- 渐变、强阴影、装饰背景、高饱和色块是否干扰工作内容。

## 5. 字体与可读性

检查：

- 页面标题、分区标题、正文、表格文字、标签、辅助说明、数字是否层级清楚。
- 运营和后台页面不使用 hero 级标题或营销式文案塞进紧凑面板。
- 长文本是否有明确处理：换行、省略、tooltip、popover、抽屉、详情页或列展开。
- 按钮文字、表头、标签、弹窗标题、字段 label 是否不溢出容器。
- 数字、日期、百分比、金额、数量、状态是否便于比较。
- 高密表格对齐是否一致；可比较数字列应统一对齐。
- 辅助说明是否短且靠近对应控件。

## 6. 交互直观性

检查：

- 主操作是否明显，次要和低频操作是否降级。
- 按钮文案是否描述结果，而不是只有模糊动词。
- 纯图标按钮是否有 tooltip 或足够上下文。
- 搜索、筛选、重置、保存、取消、删除、导入、导出、排序、分页、列设置是否出现在用户预期位置。
- 表单是否有 label、必要 placeholder、校验、提交反馈和恢复路径。
- 弹窗和抽屉是否有清晰标题、关闭路径、主按钮、次按钮和提交中状态。
- 标签页、步骤、筛选、分页、排序、列设置的状态保留或重置是否可理解。
- 破坏性操作是否在必要时确认或提供恢复路径。
- hover、active、focus、selected、disabled、loading、success 状态是否可见。

## 7. 交互承载最小化

使用最小且清晰的交互承载来匹配用户意图。

规则：

- 文字、状态、标签、表格行、列表项或附近已有控件能安全清晰地承载操作时，不额外加按钮。
- 不用状态文字和旁边按钮重复承载同一个动作，除非第二路径明显提升可发现性或安全性。
- 辅助列表旁边不要机械增加搜索按钮；点击状态、标签、数量、行或筛选值能直接收窄视图时，优先直接交互。
- 可点击文字必须通过链接样式、hover、cursor、tooltip 或上下文让用户知道它可点。
- 明确命令才使用按钮：保存、提交、删除、新建、导入、导出、重试、确认、取消、运行、停止。
- 低风险、高频、行内局部操作优先使用行内动作。
- 导航、筛选、钻取、展开详情目标明确时，优先使用行点击、标题点击、标签点击或状态点击。
- 低频或多选项操作放入菜单、抽屉或更多入口。
- 避免看起来可点但不可点的纯文本；避免只重复旁边文字的按钮。

抽象示例：

- 状态值含义是“查看这个状态下的内容”时，让状态文字或标签可点击，不要另加“搜索”按钮。
- 列表项用于打开详情时，让标题或整行承载打开行为，不要每行再加“打开”按钮。
- 数量用于钻取筛选列表时，让数量可点击，不要旁边再放“查看”按钮。
- 辅助列表只用于选择一个对象时，点击列表项就应选中；每行单独放“选择”按钮通常冗余。
- 表格行里多个安全文本都能进入详情时，选一个稳定目标，如名称、标题或编号，不要堆多个重复按钮。

定级：

- 冗余控件让主路径不清楚或产生误导：`P1`。
- 冗余按钮造成页面拥挤但不阻断使用：`P2`。
- 更直接的文字/状态交互能提升精致度：`P3`。

## 8. 控件必要性

检查每个可见控件：

- 它服务哪个用户意图？
- 这个意图是否高频到需要出现在当前层级？
- 是否已有其他控件承载同一意图？
- 是否能合并进已有状态、文字、行、标签、筛选、tab、菜单、抽屉或详情视图？
- 它是否因为命令、破坏性、不可逆、异步或权限敏感而必须使用按钮？
- 文案是否足够具体，能预测点击结果？
- 操作后是否有可见反馈？

优先：

- 合并重复筛选。
- 把低频设置折叠进设置面板、抽屉或更多菜单。
- 安全时，把重复行按钮改为点击标题、状态或标签。
- 每个区域只保留一个主命令。
- 不可用操作按项目习惯隐藏，或置灰并说明原因。

避免：

- 为了显得功能完整而存在的假完整控件。
- 只复述旁边文字的按钮。
- 到处重复“搜索 / 查看 / 详情 / 选择 / 打开 / 管理”按钮，却没有任务优先级。
- 状态本身已经可读且适合直接交互，却再单独放控制按钮。
- 暴露作用不清楚或过低频的开关。

## 9. 状态完整性

检查：

- 默认态
- 加载态
- 空数据态
- 筛选无结果态
- 错误态
- 禁用态
- 选中态
- hover 和 focus 态
- 提交中或运行中态
- 成功反馈
- 长文本和极端值
- 无权限、不可用或权限受限状态
- 慢网络或请求失败状态

状态规则：

- 空状态说明为什么为空，以及用户下一步能做什么。
- 筛选无结果状态提示清空或调整筛选。
- 错误说明发生了什么、是否能重试、用户如何恢复。
- 加载占位尽量保持布局尺寸。
- 禁用原因不明显时，应提供原因。

## 10. 动画与反馈

检查：

- 动画是否说明状态变化，而不是装饰页面。
- 动画时长是否适合高频工作。
- 加载、骨架、过渡、展开收起、弹窗、抽屉、菜单是否不闪烁。
- hover、加载、切换 tab、筛选、刷新时是否没有布局跳动。
- 高频工作界面是否避免吸引注意力的动效。
- 反馈是否不会阻塞无关操作，除非动作确实必须阻塞。

## 11. 说明与指引

检查：

- 必要说明是否靠近相关控件。
- 复杂控件是否有简短 helper text、tooltip、hint 或示例。
- 首次使用概念是否有帮助入口，但不把文档铺满页面。
- 空状态、错误、破坏性操作、不可逆操作、权限受限状态是否解释清楚。
- 不用长文案弥补命名、布局或交互设计的问题。
- 避免用可见文字解释显而易见的 UI 行为。

## 12. 响应式与视口

尊重项目目标：

- 桌面后台或数据系统只检查声明的桌面宽度和最小合理桌面宽度。
- 项目没有移动端目标时，不输出移动端问题。
- 响应式或移动端项目要检查文字可读、点击目标、导航、表格、筛选、弹窗、抽屉、横向溢出和文字重叠。
- 响应式支持不明确时，说明假设，不输出阻塞级移动端问题。

## 13. 问题质量

每个问题必须包含：

- 优先级。
- 页面或组件位置。
- 可观察现象。
- 为什么重要。
- 具体修复方向。

不要输出：

- 没有证据的个人审美。
- 纯业务逻辑批评。
- 没有具体原因的大重做建议。
- 能通过移除、合并、折叠、让现有文字或状态直接交互解决的问题，却建议新增更多 UI。
