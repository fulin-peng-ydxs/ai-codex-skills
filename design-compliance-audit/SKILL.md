# Design Compliance Audit

将项目 DESIGN 文档及其约束的前端实现都作为审查对象：先确认文档自身完整、一致、可执行、可验证且符合项目事实，再检查实现是否完整遵循。目标不是提出新的设计偏好，而是找出“DESIGN 契约本身不成立”以及“实现和既有 DESIGN 规范不一致”的地方。

## 执行流程

1. 确定审查范围：
   - 默认同时审查 DESIGN 文档本身和实现遵循情况；用户明确只审查其中一项时，按指定范围执行并说明未覆盖部分。
   - 用户指定文档、模块、路由、页面文件、PR 或 diff 时，只审查指定范围内的 DESIGN 文档、前端页面和相关实现。
   - 用户未指定范围时，默认审查 DESIGN 事实源、当前项目所有可访问页面、页面注册、组件实现、布局配置、样式入口和前端验证闭环。
   - 若任务是代码审查，优先以改动文件为入口，再追踪其影响到的页面、组件、样式、测试和文档。
2. 定位 DESIGN 事实源：
   - 优先读取用户指定的 DESIGN 文档。
   - 未指定时，优先读取仓库根目录 `DESIGN.md`；再查找 `design.md`、`docs/design*.md`、`agent-works/**/page-template*.md`、组件/布局规范、主题 token 文档和项目级 AI 指令。
   - 找不到 DESIGN 文档时，不要退化成普通 UX 审查；报告“缺少 DESIGN 事实源，无法做 DESIGN 遵循检查”。
3. 审查 DESIGN 文档本身：
   - 检查适用范围、目标终端或视口、术语、规则强度、事实源优先级和维护责任是否清楚，规则之间是否冲突、重复或存在无法判断的歧义。
   - 检查 DESIGN 文件内部以及其与仓库其他相关文档之间，是否对同一术语、规则、清单、token、组件、字段、布局约束或验收要求存在重复定义或内容冗余；应保留单一权威定义，其他位置改为引用或明确的非权威摘要，目录、链接、必要上下文和不同场景的应用说明不算重复。
   - 检查文档引用的路径、组件、token、schema、配置字段、命令、marker 和其他事实源是否真实存在且与当前项目一致；发现过时内容时给出代码或文档证据。
   - 检查硬规则是否包含可识别的适用对象、触发条件、约束动作和例外边界，并能映射到实现位置、机器契约或验证方式。
   - 结合项目实际能力检查组件复用、布局、样式/token、状态、响应式、维护闭环和验证要求是否存在关键缺口；不要求补充与项目无关的模板章节。
   - 将文档自身问题单独定级和输出；除非用户要求修订，否则只给出最小修复方向，不擅自改写 DESIGN。
4. 完整抽取约束：
   - 读取 DESIGN 文档的 front matter、目录、强制规则、禁止项、组件清单、布局流程、响应式规则、维护矩阵、验证命令和机器读取区块。
   - 重点搜索并整理这些词：`必须`、`不得`、`禁止`、`统一`、`优先`、`固定`、`唯一`、`单一事实源`、`事实源`、`维护`、`同步`、`验收`、`测试`、`marker`、`token`、`组件`、`布局`、`响应式`。
5. 对照实现：
   - 检查页面文件、组件、样式、主题变量、路由/页面注册、布局声明、组件元数据、配置字段、空/加载/错误状态、弹窗/抽屉、响应式和测试是否遵循 DESIGN。
   - 能运行项目时，结合浏览器截图或交互验证；不能运行时，明确标注为静态合规审查。
6. 定级并输出：
   - DESIGN 自身的冲突、过时硬规则、失效事实源或无法执行的验证要求，若导致契约无法确定或无法验证，至少为 `P1`；局部表达或维护性缺口按影响定 `P2-P3`。
   - DESIGN 明确写了“必须 / 禁止 / 不得 / 固定 / 单一事实源 / 唯一渲染器 / 维护清单 / 验收命令”的，违反即至少 `P1`；导致页面不可用或验证失败则 `P0`。
   - DESIGN 使用“优先 / 建议 / 默认 / 应”的，违反时按影响定 `P1-P3`，但必须说明是否有合理例外。
   - 如果实现需要偏离 DESIGN，结论只能是：修实现、补 DESIGN 文档、或要求用户确认例外；不能默认放过。

## 核心原则

- DESIGN 是契约，不是参考资料。前端实现不应静默偏离 DESIGN。
- DESIGN 文档自身也必须接受审查；不能仅因它被声明为事实源，就默认其内容正确、完整或仍然有效。
- 审查重点是“是否符合已存在的设计系统”，不是重新评价页面好不好看。
- 把统一性、复用、事实源、维护闭环和验证命令作为合规问题，而不是可选优化。
- 若 DESIGN 和现有代码冲突，不因代码现状直接否定 DESIGN；先以 DESIGN 作为实现审查基准，再结合更高优先级事实源、机器契约和有效验证证据判断是修实现、更新 DESIGN 还是迁移存量实现。
- 若 DESIGN 文档缺失、过时或互相冲突，报告事实源风险，不自行编造规范。
- 文档缺口必须以项目范围、现有实现或机器契约为证据，不按通用模板机械要求章节齐全。
- 项目没有移动端目标时，不额外输出移动端合规问题；DESIGN 声明响应式或移动端目标时必须检查。

## 必读清单

执行审查时读取 `references/compliance-checklist.md`，并按其中的规则抽取和对照 DESIGN 约束。

## 优先级

- `P0`：文档或实现问题导致页面不可用、布局崩坏、构建/测试失败、机器契约破坏、关键页面注册或验证闭环失效。
- `P1`：DESIGN 自身冲突、过时硬规则、失效事实源或不可执行验证导致契约无法确定或验证；或者实现违反明确硬规则、禁止项、唯一事实源、组件复用要求、维护矩阵、响应式红线或必须运行的验收命令。
- `P2`：DESIGN 局部歧义、缺少重要映射或维护要求但仍可判断；或者实现违反默认规范、优先复用、风格 token、布局建议、状态/空态/弹层规范，但未阻断使用。
- `P3`：轻微表达、偏离或登记遗漏，以及可后续收敛的非阻塞一致性风险。

## 边界

- 不做独立审美评判；审美和交互合理性问题交给页面 UX 审查。
- 不按理想化模板评价 DESIGN；只检查与项目适用范围、事实一致性、规则可执行性、维护闭环和验证能力相关的问题。
- 不在仅受托审查时直接改写 DESIGN；修订文档需要用户明确授权。
- 不审查业务算法、数据口径或领域规则，除非 DESIGN 文档把它们定义成页面交付约束。
- 不因为实现看起来可用就放过 DESIGN 违规；合规性以文档契约为准。
- 不把当前项目的组件名、框架名、业务名写成通用要求；只抽象检查方法。
