# Agent-Auto Commit Audit

## 概览

使用本技能审查近期 AI 自动开发提交，区分普通代码审查与业务闭环核查，修复安全边界内的问题，更新相关文档，并以默认简体中文生成邮件式报告。

## 工作流

1. 根据用户请求确定目标仓库；未指定时使用当前工作目录。
2. 审查前读取项目规则：`AGENTS.md`、必要时读取 `CLAUDE.md`，以及其中引用的全局或项目级 AI 协作约束。
3. 确定提交范围：
   - 用户明确指定范围时，使用用户范围。
   - 否则查找提交标题中最新出现 `Agent-Auto` 的提交，从该提交开始审查到 `HEAD`，包含该提交。
   - 如果没有找到 `Agent-Auto` 提交，则审查当前时间过去 7 天内到 `HEAD` 的提交。
4. 记录范围元信息：第一个提交时间、最后一个提交时间、提交 SHA、变更文件和项目名称。
   - 项目名称优先取 `git remote get-url origin` 最后路径段并去掉 `.git`；没有远程地址时使用仓库目录名。
5. 记录本次实际读取的约束文档、项目文档和实际调用的技能，按审查方向归类。
6. 编辑前执行 `git status --short`。保留无关用户改动，不回退自己未改的文件。
7. 按顺序执行：代码审查、业务审查、页面审查（适用时）、文档审查与更新。
8. 只自修复没有业务歧义、且不需要大范围架构调整的问题。
9. 修复后重新运行针对性验证。
10. 输出邮件式报告；只有邮件连接、收件人和用户发送授权都满足时，才按邮件工具规则发送，否则创建草稿或输出可发送正文。

常用命令：

```bash
git log --format='%H%x09%aI%x09%s' --date=iso
git log --since='7 days ago' --format='%H%x09%aI%x09%s'
git diff --name-status <range-start>^..HEAD
git diff <range-start>^..HEAD -- <path>
git show --stat --oneline <sha>
git status --short
git remote get-url origin
```

## 审查规则

### 1. 代码审查

使用 `code-reviewer` 技能；如果当前环境提供等价的 `/code-review` 命令或技能，则按实际可用入口执行。审查真实实现、对应测试代码和相关上下文，不只看 diff。

重点检查：

- correctness：行为正确性、回归和边界输入。
- maintainability/readability：分层、命名、重复代码、抽象和可读性。
- efficiency：数据量、查询复杂度、缓存、资源释放和重复计算。
- security：权限、租户/数据边界、密钥、注入、危险外部调用。
- edge cases/error handling：空值、异常、并发、幂等、补偿和失败路径。
- testability：测试覆盖是否匹配风险，测试断言是否真正覆盖变更行为。
- 实现是否符合需求、开发计划和项目/全局 AI 约束。

每个问题必须绑定具体文件、代码位置或紧凑代码区域，并说明影响和修复建议。不要把业务闭环问题混入普通代码审查。

如果没有发现问题，也要说明审查范围、精读文件、已看测试、未运行验证和剩余风险。

### 2. 业务审查

使用 `business-feature-audit` 技能的快速核查模式。不要重复普通代码 review。

按业务闭环检查：

- 入口、导航、API 可达性和触发路径。
- 端到端流程、状态流转和数据口径。
- 上下游协同。
- 权限、角色、租户归属和可见性。
- 异常补偿、幂等并发、历史兼容。
- 配置部署、可观测性、前后端语义一致性和页面业务闭环。

输出问题时按以下类型分类：

- `confirmed`：可由代码、配置、测试或文档证明的已确认问题。
- `inferred`：高度疑似但需要产品或业务确认的问题。
- `missing`：当前材料暂无法确认的风险。

### 3. 页面审查

仅当提交范围涉及前端页面、组件、样式、布局、路由、菜单、权限控件、浏览器可见流程或用户交互时执行。没有 UI 影响时，明确说明页面审查不适用及原因。

适用时依次使用：

1. `page-ux-audit`
2. `frontend-reuse-enforcer`
3. `design-compliance-audit`

对受影响路由执行浏览器自动化验证。优先使用项目指定的自动化端口和启动规则。至少检查主流程、可触达的空/加载/错误状态，以及受影响页面的响应式布局。

### 4. 文档审查与更新

只根据实际完成和已验证结果更新文档。不要把未完成、未验证、未确认内容写成事实。

检查以下文档是否需要更新：

- `requirement.md`：需求范围、需求编号、建议实现处理状态、验收标准和遗留确认项。
- `plan.md`：任务状态、实际落地情况、需求映射、验证计划、风险、假设和待确认项。
- 对应 `agent-works/{feature-slug}/` 下的开发留痕、业务核查、测试记录和阶段总结。
- SQL、seed、权限、菜单、配置、环境变量、脚本、`README.md`、`DESIGN.md`、`AGENTS.md`、`CLAUDE.md` 等稳定事实文档。
- 模块或功能关联的技术架构文档。

如果存在总需求文档，并且有对应开发阶段章节，将本阶段开发情况、验证情况、相关需求/开发/留痕/核查文档路径回写到对应阶段。

如果某类文档无需更新，明确说明“不需要更新”的原因。更新完成后，再检查代码、需求、计划、留痕和业务核查之间是否存在状态不一致。

## 修复规则

对已确认、非业务歧义、局部且低风险的问题进行自修复。例如：按既有模式明显缺失的权限守卫、空值边界、参数映射错误、路由登记遗漏、文档状态过期、由本范围改动导致的测试失败、明显 UI 回归。

不要自修复需要业务选择、产品范围调整、数据迁移策略、权限模型重设、大范围架构重构、不可逆数据操作、生产凭据或外部系统动作的问题。这类问题列为待修复，并给出推荐方案。

编辑前说明准备修改的文件。编辑后运行聚焦验证：

- 代码精读自审和相关测试代码复核。
- 受影响后端、领域逻辑或工具脚本的针对性测试。
- `frontend/src` 有变化时运行前端构建。
- 页面或交互变化时运行浏览器自动化验证。

遵守项目服务启停、数据库锁、测试账号和构建规则。

## 输出

报告默认使用简体中文。按四个方向组织：

1. 代码审查
2. 业务审查
3. 页面审查
4. 文档更新

每个方向内依次输出：

- 审查出的问题
- 已修复的问题
- 待修复问题和对应解决方案
- 引用的约束规则文档/技能

同时包含：

- 仓库、项目名称、提交范围、第一个和最后一个提交时间、变更文件摘要。
- 已运行的验证命令和结果。
- 未运行的验证及原因。
- 本次审查/修复改动的文件。
- 剩余风险。

“引用的约束规则文档/技能”要求：

- 只列本方向实际读取、实际引用或实际调用过的文档和技能，不把未使用的规则入口凑数写入。
- 文档写路径或稳定名称，技能写技能名；必要时用一句话说明引用目的。
- 同一方向同时用到全局规则、项目规则和专项技能时，分别列出。
- 如果某方向没有额外引用，写“无额外引用”，并说明原因。
- 页面审查不适用时，仍在该方向说明用于判断“不适用”的依据，例如变更文件摘要或项目规则。

默认邮件标题：

```text
Agent-Auto_Fix_{project_name}*{first_commit_time}-{last_commit_time}*{main_audit_summary}
```

默认邮件行为：

- 邮件连接可用、收件人明确、且用户已明确授权发送时，按邮件工具确认规则发送报告。
- 发送条件不完整或缺少收件人时，优先创建草稿；无法创建草稿时，在最终回复中输出可发送的邮件标题和正文。
- 不猜测收件人、凭据或发送结果。
