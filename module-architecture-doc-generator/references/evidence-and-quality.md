# 证据与质量门禁

## 证据规则

架构文档中的事实必须来自以下来源之一：

- 当前源码：路由、服务、仓库、模型、前端页面、状态管理、测试。
- 机器可读配置：包管理文件、构建文件、Docker、CI、环境示例、DDL、迁移。
- 现有权威文档：README、AGENTS、设计文档、架构文档、API 文档。
- 实际命令输出：仅在需要确认命令、版本、测试或构建时使用。

无法确认的内容写成待确认问题，不写成事实。

## 生成前检查

- 是否已有同模块文档；有则更新，不重复创建。
- 模块边界是否与代码边界、用户入口和数据所有权匹配。
- 文档目录是否已有项目约定；必须先确定输出目录和每个模块的输出文件路径。
- 需要同步 README 或索引时，只更新入口，不复制全文。

## 内容质量

必须保留：

- 当前模块定位和职责边界。
- 核心实现机制和关键代码入口。
- 主要页面/API/CLI/任务/配置入口。
- 关键流程、状态变化、上下游影响。
- 数据、权限、安全、并发、缓存、外部依赖等硬边界。
- 当前真实风险及建议。

必须删除：

- 迁移过程、阶段流水、旧计划、临时背景。
- 与当前代码不一致的旧事实。
- 无证据的猜测。
- README 或需求文档的大段复制。
- 与目标模块无关的其它模块细节。

## 风险分级

推荐风险表字段：

| 等级 | 风险点 | 影响 | 建议 |
|------|--------|------|------|

等级按实际影响分：

- 高：可能造成安全、数据损坏、资金/业务关键错误、严重不可用。
- 中：可能造成结果偏差、性能退化、维护困难、局部不可用。
- 低：已知限制、体验问题、可接受降级或后续优化项。

## 落地检查

写文件后检查：

- 文件落在已确认的架构输出目录中。
- 链接和相对路径有效。
- Mermaid 语法结构合理。
- 表格没有过宽到不可读。
- 每个模块文档职责单一。
- README/索引只增加入口，不与模块文档重复。
- 没有把来源项目专有业务内容带入其它项目。
