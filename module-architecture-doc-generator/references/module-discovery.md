# 模块识别

## 识别原则

模块边界来自项目事实，而不是目录名本身。优先选择能独立解释一条业务闭环、一个用户入口、一组 API、一个数据所有权范围或一个平台基础能力的边界。

## 输入来源

生成前至少检查：

- `README.md` 的功能模块、工作流和文档入口。
- `AGENTS.md` 中的核心契约、目录结构、运行和权限边界。
- 现有 `agent-works/architecture/` 或项目内已存在的其它架构文档目录。
- 后端路由、服务、仓库、模型、DDL、CLI。
- 前端路由、页面、组件、状态管理、权限指令。
- 测试目录和测试命名。
- 配置、启动脚本、任务调度、迁移和部署文件。

## 候选模块类型

- 业务模块：围绕用户目标和页面/API 闭环，例如订单、回测、数据管理、报表。
- 平台模块：跨业务复用的底座，例如鉴权、任务调度、配置、日志、缓存、部署。
- 数据模块：围绕数据采集、清洗、存储、同步、质量、迁移。
- 前端模块：围绕页面体系、设计系统、布局框架、组件库。
- 集成模块：围绕外部服务、第三方 API、消息队列、支付、通知。
- 开发运维模块：围绕服务生命周期、CI/CD、环境、发布、监控。

## 拆分规则

拆成多篇文档的信号：

- 有不同用户入口或角色。
- 有不同 API 前缀、CLI 命令或任务类型。
- 有不同数据表或数据所有权。
- 风险和验收边界明显不同。
- 文档超过可读范围，读者必须跳着找信息。

合并成一篇文档的信号：

- 多个目录只是同一模块的分层实现。
- 页面、API、服务和数据高度耦合，无法独立解释。
- 子能力没有独立使用入口和风险边界。

## 命名规则

- 文件名使用稳定 kebab-case，例如 `auth-rbac.md`、`data-management.md`。
- 标题使用业务或技术读者能理解的中文名称。
- 不使用临时版本号、阶段名、人名或日期作为模块文档名。

## 输出路径

按以下顺序确定输出目录：

1. 用户明确指定的目录。
2. 项目已有架构目录，优先识别 `agent-works/architecture/`，其次使用项目已经存在且明确承载架构文档的目录。
3. 如果用户未指定目录且项目没有现成架构目录，创建并使用 `agent-works/architecture/`。

输出文件路径为 `<输出目录>/<module-slug>.md`。如果同一模块已有文档，更新已有文档原路径，不另建新文件。

默认兜底路径是 `agent-works/architecture/`，模块架构文档输出到 `agent-works/architecture/{module-slug}.md`。
