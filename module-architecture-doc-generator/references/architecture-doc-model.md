# 架构文档模型

## 基础结构

模块架构文档推荐由以下模块组成，按项目实际取舍：

1. `标题与定位`
   - 用一句话说明模块解决什么问题、服务谁、为什么存在。
   - 可加 1 个短引用块写模块边界或核心价值。

2. `核心技术`
   - 写关键代码入口、算法、框架机制、数据结构、状态机、任务模型、缓存、并发、鉴权等。
   - 每条尽量带路径、函数、类、表或配置证据。

3. `业务架构 / 系统架构`
   - 说明模块内部组成、上下游关系、数据流、调用链。
   - 关系复杂时使用 Mermaid；简单模块用表格即可。

4. `使用说明`
   - 面向真实使用者描述页面、API、CLI、任务、配置或运维入口。
   - UI 模块写页面入口和操作区；后端模块写 API/CLI；平台模块写运维和开发使用。

5. `业务流程 / 运行流程`
   - 写关键流程的步骤和状态变化。
   - 优先覆盖新增、查询、执行、变更、失败恢复、异步任务、权限校验等路径。

6. `数据、权限与配置`
   - 仅当模块涉及持久化、数据归属、权限、配置生效、外部服务时写。
   - 写表、字段、owner/tenant、可见性、资源码、配置路径和环境变量。

7. `与其它模块的关系`
   - 写上游输入、下游输出、共享能力和影响范围。
   - 适合用表格表达。

8. `风险事项`
   - 写当前存在的风险、影响和建议。
   - 风险必须是当前事实，不写泛泛提醒。

## 当前项目文档可借鉴的通用模式

- `核心技术` 不只列技术栈，而是解释关键实现机制和代码位置。
- `业务架构` 连接页面、API、服务、数据表和下游模块。
- `页面使用说明书` 面向实际操作者，说明每个入口能做什么、如何判断结果。
- `业务流程` 写端到端状态变化，而不是散点说明。
- `风险事项` 用等级、风险点、影响、建议四列，便于开发和产品评估。
- 平台型模块把“运维与开发说明书”替代“页面使用说明书”。

这些是结构方法，不是领域内容；不要复制来源项目的模块名、领域口径、端口、账号或业务规则。

## 条件章节

按模块事实决定是否加入：

- `页面使用说明书`：存在用户界面或管理后台。
- `运维与开发说明书`：平台、部署、任务、配置、调度、基础设施模块。
- `API 契约`：模块主要以 API/SDK 对外。
- `数据模型`：模块有独立表、文档结构、缓存键、消息结构或文件格式。
- `权限与安全边界`：模块涉及登录、RBAC、多租户、资源归属、审计或敏感操作。
- `性能与容量`：模块存在并发、队列、缓存、长任务、大数据量或成本限制。
- `扩展方式`：模块经常要新增类型、插件、指标、页面、任务或外部适配器。
- `验收与测试入口`：模块有关键测试、扫描脚本、CI、截图或手工验收流程。

## 写作边界

- 不写未实现路线图，除非用户明确要求独立规划文档。
- 不把需求、开发计划、PR 记录或历史修复过程复制进架构文档。
- 不把所有源码路径堆成清单；只写理解模块必须知道的入口。
- 不在文档中编造 API、表、页面或配置项。
