# Session Code Review Test

## Overview

这个技能专门针对“当前会话中改过的代码”做会话级代码审查和最小充分验证。

本文中的“当前会话改动”统一指当前会话内由助手生成、新增、修改或删除的代码变更，不包含审查阶段尚未落地的假设性修改。

目标不是做全仓库审计，而是快速回答以下问题：

1. 当前会话改动的业务逻辑是否满足当前任务意图。
2. 当前会话改动是否复用了项目内已有实现，代码风格是否与项目统一。
3. 当前会话改动的注释是否达到项目要求的说明度和可维护性标准。
4. 当前会话改动是否引入编译、构建、类型或基础运行错误。
5. 如果前后端都有改动，接口联调是否存在风险或不一致。
6. 当前会话改动是否存在明显性能问题。

默认将 `references/review-checklist.md` 作为统一检查清单使用。

## Trigger Guidance

以下表达应优先触发本技能或与本技能高度相关：

- “围绕当前会话改过的代码做 review”
- “检查当前会话改动有没有问题”
- “看这次 AI 写的代码靠不靠谱”
- “顺手帮我审查并测一下这次修改”
- “检查当前会话代码的编译、联调和性能风险”

如果用户要按需求文档和开发计划做正式验收，应优先使用 `code-review-improve`。
如果用户重点诉求是“按需求文档和开发计划做正式验证补测”，应优先使用 `requirements-plan-test-verifier`。

## Workflow

1. 锁定当前会话中的改动范围。
2. 理解当前会话改动的任务目标和预期行为。
3. 按检查清单做代码审查，并执行与改动匹配的最小充分测试。
4. 若问题可最小修复则修复，否则明确给出问题和建议。
5. 按固定格式输出审查与测试结论。

## Step 1: Lock Change Scope

必须先锁定“当前会话中改过的代码”范围。

优先级如下：

1. 用户明确指定的文件、模块、函数、页面、接口。
2. 当前会话中由助手新增、修改、删除过的文件。
3. 当前会话直接提到并与本次实现强相关的调用链上下游文件。

规则：

1. 默认不要扩大全项目扫描。
2. 若范围仍不清楚，可结合当前会话中的编辑记录、变更文件和相关依赖做最小扩展。
3. 若无法可靠识别当前会话改动范围，再向用户确认。

执行时先输出一句范围说明，例如：

- “本次围绕当前会话改动的 `A/B/C` 文件做审查和测试。”

## Step 2: Understand Task Intent

围绕当前会话上下文理解当前会话改动原本要解决什么问题。

至少要明确：

1. 当前任务目标是什么。
2. 代码期望实现什么行为。
3. 哪些模块、接口、页面或流程受影响。
4. 哪些行为属于本次应改，哪些不属于本次范围。

审查业务逻辑时，以“当前会话中的开发目标”作为基线，而不是自行发挥。

## Step 3: Review And Test The Changed Code

按 `references/review-checklist.md` 做审查。

本步骤同时完成两件事：

1. 对当前会话改动做代码审查。
2. 执行与当前会话已变更代码相匹配的最小充分验证。

重点检查以下六类问题：

1. 业务逻辑是否满足：
- 是否完成当前任务目标。
- 是否存在逻辑遗漏、错改、误改、边界缺失。
2. 复用性与风格统一：
- 是否优先复用项目内已有工具类、公共组件、公共方法、统一封装和相似实现。
- 是否保持接口响应格式、异常处理、命名、分层、页面交互风格一致。
3. 注释完整性与说明度：
- 类、字段、方法、核心业务逻辑是否补齐了必要注释。
- 注释是否说明业务背景、设计意图、边界处理和兼容原因，而不是空话或逐句翻译代码。
4. 编译/构建/类型风险：
- 是否存在明显编译报错、构建报错、类型错误、引用错误、路径错误。
5. 前后端接口联调风险：
- 若前后端都有改动，检查接口路径、方法、参数、字段、空值语义、错误处理是否一致。
6. 性能问题：
- 是否存在明显重复查询、重复计算、无效渲染、无效 IO、过度轮询或高频无意义开销。

执行要求：

1. 先找真实问题，再给最小改动建议。
2. 审查要围绕“当前会话改动是否靠谱”，不是追求泛化完美。
3. 对每个问题尽量落到具体文件、函数、接口或调用链。
4. 编译、构建、类型检查和最小充分测试是本步骤的必执行项，不单独拆成可选阶段。
5. 验证范围由当前会话已变更代码决定，不默认跑全量重测试。

验证要求按改动类型选择：

1. 后端改动：至少执行相关编译、局部测试、关键接口或关键逻辑验证。
2. 前端改动：至少执行相关类型检查、局部构建、页面或交互验证。
3. 前后端联动改动：至少检查一条成功链路和一条失败或边界链路。
4. 纯重构或纯样式类改动：选择能证明“未破坏原行为”的最小验证。
5. 若受环境限制无法执行，必须在最终结论中明确写出未执行原因和残留风险。

## Step 4: Fix Or Report

发现问题后按以下原则处理：

1. 若问题可以最小改动修复，直接修复并重新验证。
2. 若问题涉及较大业务调整、结构重构或超出本次任务范围，先向用户确认。
3. 若问题无法在当前环境验证，先明确风险，不硬下结论。

## Step 5: Output Format

按固定顺序输出：

1. 改动范围
- 本次审查和测试覆盖哪些文件、模块、接口、页面。

2. 审查结论概览
- 业务逻辑：是否满足
- 复用与风格：是否统一
- 注释规范：是否达标
- 编译/构建/类型：是否存在问题
- 前后端联调：是否存在问题
- 性能：是否存在问题

3. 问题清单
- 按严重程度列出问题
- 每项写清位置、问题类型、影响、建议

4. 最终结论
- 是否可继续交付
- 是否需要先修复
- 残留风险有哪些

## Execution Rules

1. 默认只围绕当前会话改过的代码审查和测试，不扩大全仓。
2. 输出优先呈现问题和风险，再给总结。
3. 发现前后端都有改动时，接口联调检查默认升级为必查项。
4. 若存在编译、构建、类型错误，这类问题优先级高于风格建议。
5. 若发现新增或修改代码缺少必要注释，或注释说明度明显不足，应作为明确问题列出，而不是只给风格建议。
6. 若未发现明显问题，明确写“未发现明显问题”，同时说明测试边界和残留风险。

## Output Template

优先直接套用以下模板：

```md
**改动范围**
- 文件/模块：<当前会话改动范围>
- 目标：<当前会话改动想解决什么问题>

**审查结论概览**
- 业务逻辑：<满足 / 不满足 / 部分满足>
- 复用与风格：<统一 / 不统一 / 部分统一>
- 注释规范：<达标 / 不达标 / 部分达标>
- 编译/构建/类型：<有问题 / 未发现明显问题>
- 前后端联调：<有问题 / 未发现明显问题 / 不涉及>
- 性能：<有问题 / 未发现明显问题>

**问题清单**
1. <问题标题>
- 类型：<业务逻辑 / 复用 / 风格 / 注释 / 编译 / 类型 / 联调 / 性能>
- 位置：<文件 / 函数 / 接口 / 页面>
- 影响：<影响说明>
- 建议：<最小改动建议>

2. <问题标题>
- 类型：<...>
- 位置：<...>
- 影响：<...>
- 建议：<...>

**最终结论**
- 交付判断：<可交付 / 建议修复后再交付 / 无法确认>
- 残留风险：<如无则写“无”>
```
