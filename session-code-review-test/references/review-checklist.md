# Review Checklist

这个清单用于 `session-code-review-test`，围绕“当前会话中改过的代码”做快速审查和测试。

## A. Business Logic

1. 当前任务目标是否实现。
2. 关键流程是否按预期工作。
3. 边界条件、空值、失败路径是否处理。
4. 是否出现 AI 顺手多改、乱改或偏离原目标。

## B. Reuse And Style

1. 是否优先复用现有工具类、公共组件、公共方法、统一封装。
2. 是否重复造轮子。
3. 命名、分层、返回格式、异常处理、页面交互是否与项目一致。

## C. Comment Quality

1. 类是否说明职责、使用场景、与上下游模块或对象的关系。
2. 字段是否说明字段含义、业务语义、取值约束、特殊约定或单位。
3. 方法或函数是否说明用途、核心参数、返回值、前置条件、关键副作用或异常语义。
4. 核心业务逻辑代码块是否说明业务意图、执行流程、关键判断依据、边界处理、兼容原因。
5. 注释是否优先解释业务背景、设计意图和约束条件，而不是逐句翻译代码。
6. 是否存在“查询数据”“保存信息”这类无信息量方法注释。
7. 是否存在只写字段字面意思、未补充业务用途和状态语义的字段注释。
8. 兼容旧逻辑、临时兜底、协议映射、历史数据修正、异常绕行等代码是否明确说明原因。
9. 注释是否紧贴声明或逻辑本身，保持结构化和可维护，而不是模板化凑数。

## D. Build And Type Safety

1. 是否存在编译错误。
2. 是否存在构建错误。
3. 是否存在类型错误、引用错误、路径错误。
4. 是否存在明显会导致运行失败的基础问题。

## E. FE/BE Integration

当前后端都有改动时检查：

1. 接口路径和方法是否一致。
2. 参数、字段、类型是否一致。
3. 成功态、失败态、空值语义是否一致。
4. 前端调用和后端实现是否匹配。

## F. Performance

1. 是否存在明显重复查询、重复计算、无效 IO。
2. 是否存在无意义重渲染、无意义请求、无意义轮询。
3. 是否引入明显可见的性能回退。

## G. Test Evidence

1. 是否执行了与本次改动匹配的最小充分测试。
2. 是否明确记录通过项、失败项、未执行项。
3. 是否说明未执行原因和残留风险。
