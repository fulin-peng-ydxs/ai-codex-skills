# Backend Memory Risk Report

## 概览

使用这个技能分析项目后端代码中可能存在的内存溢出、内存泄漏和持续内存增长风险，并输出完整报告。

默认做代码级风险分析，而不是伪装成运行时已确认结论。若没有 heap dump、profiler 输出、GC 日志或生产监控指标，应把结论表述为“疑似风险”或“潜在风险”，并给出置信度和后续验证建议。

详细风险模式和报告规则见 `references/memory-risk-checklist.md`。

## 工作流

1. 锁定后端分析范围。
2. 识别后端运行形态和长生命周期组件。
3. 检查高风险内存模式。
4. 按证据强度分级和判定。
5. 输出并落盘完整风险报告。

## 第一步：锁定范围

先识别需要分析的后端模块。

优先顺序：

1. 用户明确指定的后端模块或服务路径。
2. 常规后端目录，例如 `src/main/java`、`server`、`service`、`listener`、`scheduled`、`config`、`util`、`dao`、`repository`。
3. 与运行时强相关的代码，例如调度任务、监听器、缓存、MQ 处理器、批处理任务、上传导出逻辑、集成层。

如果仓库是前后端混合项目，除非为了理解后端引用链，否则不要扩展到前端代码。

## 第二步：识别运行形态

在找泄漏前，先识别这个后端里“什么对象可能被长期持有”。

至少要明确：

1. 语言和运行时：
- Java/Spring Boot、Node.js、Go、Python 或其他。
2. 长生命周期组件：
- 单例服务、静态持有对象、缓存、调度器、监听器、线程池、MQ 消费者、WebSocket/会话管理器。
3. 数据密集型路径：
- 大查询、导入导出、文件处理、JSON 序列化、批量聚合、内存内 Join、流式缓冲。
4. 外部集成点：
- Redis、MQ、MQTT、WebSocket、Feign/HTTP 客户端、TDengine/MySQL、文件存储、视频流、规则引擎。

对于 Java/Spring 项目，优先检查单例 Bean、静态集合、`ThreadLocal`、线程池队列、定时任务、监听器和大集合处理路径。

## 第三步：检查内存风险模式

以 `references/memory-risk-checklist.md` 作为主检查清单。

重点关注：

1. 无边界对象持有：
- 静态 `Map/List/Set`、单例缓存、本地内存注册表、会话映射、持续增长的队列。
2. 长生命周期引用泄漏：
- 未清理的监听器、回调注册表、未移除的 `ThreadLocal` 值、从不清理的订阅关系。
3. 大对象和缓冲风险：
- 大结果集一次性加载进内存、反复构建 `List`、`ByteArrayOutputStream`、大 JSON 转换、导入导出缓冲。
4. 调度与异步堆积：
- `@Scheduled` 任务不断累积状态、重试队列、延迟任务、没有背压的执行器队列。
5. 间接引发内存压力的资源泄漏：
- 流未关闭、HTTP 响应未释放、文件句柄或驱动资源持有时间过长。
6. 高频对象创建导致的内存尖峰：
- 热路径反复创建大对象、大负载多次拷贝、多轮内存转换。

搜索和判断时，优先使用直接代码证据，不要泛泛而谈。

Java/Spring 项目可优先搜索：

- `static `
- `ThreadLocal`
- `ConcurrentHashMap`
- `new HashMap`
- `new ArrayList`
- `@Scheduled`
- `ExecutorService`
- `ScheduledExecutorService`
- `newFixedThreadPool`
- `newCachedThreadPool`
- `BlockingQueue`
- `ByteArrayOutputStream`
- `InputStream`
- `OutputStream`
- `EasyExcel`
- `collect(Collectors.toList())`
- `subList`
- `addListener`
- `register`
- `subscribe`
- `mqtt`
- `websocket`
- `cache`

## 第四步：分级与判定

不要把每一处可疑代码都直接说成“内存泄漏”，要先分级。

统一使用以下分级：

1. `High Risk`
- 有明确代码证据表明对象会长期存活或无限增长。
2. `Medium Risk`
- 有明显增长风险，但触发条件或清理逻辑还不够明确。
3. `Low Risk`
- 模式值得关注，但当前代码证据偏弱。
4. `Needs Runtime Evidence`
- 风险合理存在，但需要 heap dump、profiler、GC 日志或生产监控进一步确认。

每条风险都必须说明：

1. 可能被持有的对象或数据是什么。
2. 为什么它会持续增长或存活过久。
3. 在什么触发条件下会变得危险。
4. 代码里有什么证据。
5. 首先应该改什么。

## 第五步：输出并落盘报告

输出必须是完整报告，而不是零散结论。

报告必须包含：

1. 分析范围
- 检查了哪些后端模块。
2. 运行形态摘要
- 语言、框架、长生命周期组件、主要风险面。
3. 风险清单
- 按风险等级排序。
4. 总体判断
- 是否存在明显 OOM/泄漏风险。
5. 处理进度
- 每条风险的当前状态。
6. 下一步建议
- 代码修复、运行时验证和监控建议。

默认报告输出路径：

- `docs/check/memory/{日期}.md`

规则：

1. 若 `docs/check/memory` 不存在，则逐级创建。
2. `{日期}` 默认使用当天日期，建议格式为 `YYYY-MM-DD`。
3. 生成当天报告前，先查找 `docs/check/memory` 下时间上最新的一份已有报告，并将其作为本次报告生成基础。
4. 对最新已有报告中的风险项，结合当前代码现状更新其处理进度、风险等级、代码证据和建议。
5. 若本次扫描发现之前报告中没有的新风险，则在本次报告中新增这部分风险项。
6. 若同日期报告已存在，则在该文件上继续增量更新，不要盲目覆盖有效内容。
7. 除非用户另有要求，最终结果应以报告文件为主，而不是只在聊天中给摘要。

## 执行规则

1. 优先使用具体代码证据，而不是宽泛架构猜测。
2. 没有运行时证据时，必须写成“疑似风险”或“潜在风险”。
3. 优先关注长生命周期对象持有，而不是短生命周期分配噪声。
4. 对 Java/Spring 后端，先检查调度任务、监听器、缓存、执行器、单例服务和大集合处理。
5. 当未发现明显风险时，也要明确写出“暂未发现明显风险”和剩余盲区。
6. 每条风险必须带处理进度字段，统一使用以下状态之一：
- `已处理`
- `未处理`
- `业务需要`
- `不重要`
- `需运行时验证`
7. 默认不要把每天的报告写成相互孤立的快照，而要基于“上一份最新报告 + 本次代码现状”形成连续演进记录。
