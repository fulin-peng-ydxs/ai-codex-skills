# Business Feature Audit

对本次开发涉及的代码做业务闭环核查。目标不是重复做普通代码审查，而是判断这次改动放回业务模块和完整链路后，整体功能是否成立，是否有未充分考虑的问题。

详细核查维度见 `references/audit-dimensions.md`，文档模板见 `references/report-template.md`。

## 适用范围

在以下场景使用本技能：

- 用户要求核查“这次开发对应的整个业务功能是否合理”。
- 用户要求检查本次改动在某个具体业务模块中的完整性、闭环性和遗漏风险。
- 用户要求根据当前改动输出业务风险结论。
- 用户要求输出业务功能核查文档，而不是只做代码层面的 review。

本技能只做业务核查和业务核查文档生成，不修改业务代码。

## 工作模式

根据用户意图选择一种模式：

### 快速核查模式

适用于用户只要求“看看这个业务功能有没有问题”“核查本次改动是否闭环”“这个技能有没有风险结论”等场景。

- 可以在用户未明确模块时，先基于会话上下文、Git diff、文件路径、类名、接口名和已有文档反推候选业务模块。
- 候选模块只能写成“推断模块”或“待确认模块”，不得写成用户已确认事实。
- 可以直接输出初步业务风险结论、证据、疑点和需要确认的问题。
- 不得创建或修改 `agent-works` 文档。
- 如果候选模块无法合理推断，提示用户补充业务模块后停止。

无法推断模块时使用固定提示语：

```text
请先明确这次要核查的具体业务模块，例如“隐患整改模块”或“告警处置模块”。当前改动无法可靠反推出业务模块，未指定具体模块前我不会继续做业务核查。
```

### 正式产档模式

适用于用户明确要求“生成业务核查文档”“输出到 agent-works”“更新 business-audit.md”等场景。

- 写入或更新文档前，必须确认具体业务模块、文件范围、业务功能拆分和目标路径。
- 用户未确认前，可以分析代码、形成核查方案和初步结论，但不得创建或修改 `agent-works` 文档。
- 只有用户明确表示确认、同意、继续、可以生成等含义后，才执行写入。

## 模块边界

可接受示例：

- 隐患排查模块
- 隐患整改模块
- 告警处置模块
- 风险分级管控模块
- 双重预防大屏统计模块

模块确认规则：

- 用户已明确模块时，以用户模块为边界。
- 用户未明确模块但当前改动能可靠反推时，可以进入快速核查模式，并标记为“推断模块，待用户确认”。
- 用户未明确模块且无法可靠反推时，必须提示用户补充模块并停止。
- 正式产档模式下，模块必须经用户确认；不得仅凭目录名、类名、接口名直接写入报告。

## 信息来源

按以下顺序收集证据，结论必须能从证据中推导，并标明证据强度：

1. 用户在当前会话中明确给出的业务模块、核查目标和补充说明。
2. 当前会话中实际新增或修改的业务文件。
3. Git 工作区改动：`git status --short`、`git diff --stat`、必要时查看具体 `git diff -- <file>`。
4. 项目已有需求、开发计划和 `agent-works` 文档：优先查找项目内 `./agent-works/`，必要时再查找工作目录中的相关计划。
5. 与本模块直接相关的代码上下文：控制器、服务、Mapper、页面、组件、接口封装、路由、配置、定时任务、消息消费、统计查询等。

不要把未查看、未验证、无法从当前证据支撑的内容写成事实。

证据强度：

- `confirmed`：来自已查看代码、Git diff、现有需求文档、开发计划或用户明确说明，可直接支撑结论。
- `inferred`：从命名、调用链、目录结构、相邻代码或常见业务模式推断，需要标记推断依据。
- `missing`：当前缺少配置、数据、外部系统契约、联调结果或业务规则，不能直接下结论。

证据冲突处理：

- 用户明确说明与代码表现冲突时，分别记录“用户说法”和“代码证据”，不要强行合并。
- 需求文档与代码实现冲突时，以冲突本身作为待确认问题或已确认偏差。
- 无法证明实际线上配置、历史数据、外部接口行为时，不得写成已确认。

## 核查原则

1. 以用户确认模块或明确标记的推断模块为边界。
2. 以“本次开发触达的业务功能”作为核查单位，而不是按文件逐个点评。
3. 问题必须绑定完整业务链路中的位置和影响，不能只给局部代码意见。
4. 结论必须区分：
- 已确认问题
- 待业务确认问题
- 信息不足，暂无法确认
5. 纯代码缺陷只能作为业务影响的证据，不作为输出主体。

## 业务功能拆分规则

从改动代码反推业务功能时，优先按业务目标拆分，而不是按文件拆分。

拆分优先级：

1. 用户或需求文档明确列出的验收项、功能点、流程节点。
2. 独立业务目标，例如新增隐患、整改闭环、复查确认、统计看板。
3. 独立状态闭环，例如提交、派发、整改、验收、关闭。
4. 独立用户入口或外部入口，例如页面操作、接口调用、导入、定时任务、消息消费。
5. 独立下游影响，例如待办、消息、统计、导出、第三方同步。

合并规则：

- 多个文件共同完成同一个业务目标时，合并为一个业务功能。
- 同一接口同时影响主流程和统计/消息等旁路输出时，主流程作为核心功能，旁路输出作为该功能的链路环节。
- 如果两个功能有独立入口、独立状态闭环或独立上线验收标准，应拆分。

## 输出位置与命名

正式产档时，统一输出到项目根目录下的功能目录：

```text
./agent-works/{feature-slug}/business-audit.md
```

规则：

- 每个功能需求一个目录。
- 目录内固定使用 `business-audit.md` 作为业务核查文档文件名。
- 文档标题使用中文业务功能名称。
- `{feature-slug}` 使用小写 kebab-case，例如 `hazard-rectification-flow`。
- 如果目标 `business-audit.md` 已存在，优先更新已有文件；更新时保留仍然准确的历史内容，删除或改写过期内容。
- 不为同一功能创建多个同类核查文档。

## 确认流程

写入或更新 `agent-works` 文档前，必须先向用户确认核查方案。

确认内容必须包含：

- 本次核查的具体业务模块。
- 本次纳入核查的文件范围：列出业务文件，说明排除的构建产物、临时文件或无关文件。
- 识别出的业务功能：每个功能的中文标题、英文目录名 `{feature-slug}`、对应文件和拆分理由。
- 已找到的需求、开发计划或已有 `business-audit.md` / `development-trace.md` 文档。
- 输出动作：标明是新建文档还是更新已有文档，以及目标路径。

确认方式：

1. 先展示“拟生成/更新的业务核查方案”。
2. 明确等待用户确认；用户未确认前不得创建或修改 `agent-works` 文档。
3. 如果用户要求调整文件范围、功能拆分、标题、目录名或目标路径，先按反馈更新方案，再次确认。
4. 只有用户明确表示确认、同意、继续、可以生成等含义后，才执行写入。

## 执行流程

1. 判断用户意图：快速核查还是正式产档。
2. 定位项目根目录，查看 `git status --short` 获取当前改动范围。
3. 结合会话上下文确认本次需要核查的业务文件，排除构建产物、依赖目录、临时文件和无关文档。
4. 确认或推断业务模块；若无法确认或推断，提示用户补充模块并停止。
5. 查找与该模块相关的需求、开发计划、已有 `agent-works` 文档和关键代码上下文。
6. 从改动代码反推本次触达的业务功能，并按“业务功能拆分规则”分组。
7. 围绕每个业务功能构建完整业务链路：入口、核心处理、状态流转、异常分支、对外输出、上下游影响、补偿与运营落地。
8. 结合 `references/audit-dimensions.md` 逐项核查合理性和遗漏风险，并标明证据强度。
9. 快速核查模式：直接输出初步业务风险结论、已确认问题、待确认问题、无法确认风险和建议下一步。
10. 正式产档模式：形成拟生成/更新方案，并按“确认流程”向用户确认。
11. 用户确认后，生成或更新 `agent-works/{feature-slug}/business-audit.md`。
12. 完成后汇报生成或更新的文档路径、覆盖的业务功能、主要证据来源、结论摘要和剩余盲区。

## 输出约束

- 文档内容使用中文。
- 未经用户确认核查方案，不得创建或修改 `agent-works` 文档。
- 快速核查模式输出必须标明模块是“用户确认”还是“推断待确认”。
- 不复制敏感配置值；涉及配置时只写文件位置、用途和环境差异。
- 不夸大结果，不把推测写成事实。
- 未执行的验证、测试、联调或线上行为，不得写成“已确认”。
- 当未发现明显问题时，也要写清剩余盲区和未覆盖范围。
- 本技能不替代代码 review；如发现纯代码缺陷，可作为业务影响的补充证据，而不是输出主体。
- 本技能不审查页面排版、颜色、动画、风格统一、DESIGN 遵循或前端复用治理；页面体验交给 `page-ux-audit`，DESIGN 合规交给 `design-compliance-audit`，前端重复实现和抽象边界交给 `frontend-reuse-enforcer`。
