# business-audit.md 模板

正式产档模式默认使用以下结构生成或更新 `./agent-works/{feature-slug}/business-audit.md`。

快速核查模式不写文件，但应复用“核查结论、已确认问题、待业务确认的问题、暂无法确认的风险点、建议下一步”的结构输出初步结论，并标明模块是否为推断。

```markdown
# <业务功能名称>

## 核查范围

- 业务模块：<用户确认的具体模块；如来自推断，写“推断模块，待确认”>
- 本次核查目标：<用户要求核查的目标>
- 文件范围：
  - `<文件 1>`
  - `<文件 2>`
- 证据来源：
  - `<来源>`：<confirmed / inferred / missing>，<简要说明>
  - `<来源>`：<confirmed / inferred / missing>，<简要说明>

## 模块与业务功能说明

- 说明该业务模块在本次核查中涉及的具体功能点。
- 如果一个模块中包含多个业务功能，写清本次实际触达哪些功能、未触达哪些功能。
- 写清业务功能拆分依据：业务目标、状态闭环、用户入口、上线验收项或下游影响。

## 本次改动对应的业务链路

### 1. 入口

- <页面入口 / 接口入口 / 任务入口 / 消费入口>

### 2. 核心处理

- <核心服务逻辑、状态变更、规则判断、数据更新>

### 3. 对外输出或下游联动

- <列表、详情、统计、导出、待办、消息、缓存、审计、第三方联动等>

### 4. 异常或补偿路径

- <失败处理、回滚、补偿、人工介入方式>

## 核查结论

- 结论等级：<合理 / 基本合理但有遗漏 / 存在明显闭环缺口 / 风险较高不建议直接上线>
- 证据强度：<confirmed / inferred / missing>
- 结论说明：<2-4 句，概括本次开发放回整体业务后的判断；如涉及页面，说明页面体验和专业交付是否支撑业务落地>
- 剩余盲区：<未验证的配置、数据、联调、外部系统、权限或历史兼容范围>

## 已确认问题

### 1. <问题标题>

- 严重级别：<high / medium / low>
- 证据强度：confirmed
- 所属模块：<具体模块>
- 所属业务功能：<功能名称>
- 代码证据：<文件 / 类 / 方法 / 关键逻辑>
- 问题描述：<说明哪里不合理或遗漏了什么>
- 业务影响：<会导致什么业务后果>
- 触发条件：<在什么情况下暴露>
- 修复建议：<建议优先怎么改>

> 若无，写“无”。

## 待业务确认的问题

### 1. <问题标题>

- 严重级别：<high / medium / low>
- 证据强度：<confirmed / inferred>
- 所属模块：<具体模块>
- 所属业务功能：<功能名称>
- 代码证据：<文件 / 类 / 方法 / 关键逻辑>
- 疑点描述：<代码上看到的疑点>
- 需要确认的业务规则：<需要业务或产品明确什么>
- 潜在影响：<若判断为真，会带来什么后果>
- 建议：<建议如何确认或补充>

> 若无，写“无”。

## 暂无法确认的风险点

### 1. <风险标题>

- 证据强度：missing
- 涉及位置：<文件 / 类 / 方法 / 外部依赖>
- 当前缺失信息：<缺了什么信息导致无法判断>
- 潜在风险：<若存在问题，可能影响什么>
- 建议补充材料：<配置、数据、联调记录、运行结果等>

> 若无，写“无”。

## 涉及文件

- `<文件 1>`：<职责>
- `<文件 2>`：<职责>

## 建议下一步

- <优先确认的业务规则、补充材料、联调验证、配置检查、页面体验改进或修复方向>

## 备注

- 写明不在本次核查范围内但容易被误认为已覆盖的内容。
- 写明尚未验证的外部依赖、环境条件或部署动作。
```

## 更新规则

- 若 `business-audit.md` 已存在，先保留仍然准确的业务背景、历史问题和已有结论。
- 对已失效内容进行改写或删除，不保留明显过期结论。
- 新发现的问题追加到对应章节；已解决的问题只有在用户要求保留历史演进时才单独说明。
- 不把本次未核查的部分写成“已覆盖”。
