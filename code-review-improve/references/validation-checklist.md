# Validation Checklist

这个清单用于 `code-review-improve` 的代码审查、优化与验收阶段。

使用前提：

1. 已锁定唯一需求文档。
2. 已锁定唯一开发计划。
3. 已明确本次审查范围与对应代码落点。

这个清单不是泛化的“代码健康检查”，而是围绕“需求文档 + 开发计划 + 实际代码”做一致性审查。

## A. Baseline Alignment

先检查基线是否一致：

1. 需求文档与开发计划是否匹配同一功能。
2. 开发计划中的任务拆分、步骤顺序、状态描述是否仍然有效。
3. 代码落点是否能映射到需求条目和计划步骤。
4. 若需求、计划、代码三者冲突，是否已明确记录冲突点。

## B. Requirement Coverage

逐条检查需求是否真正落地：

1. 功能点是否实现完整。
2. 业务规则、限制条件、边界条件是否实现完整。
3. 验收标准是否有对应实现或验证证据。
4. 是否存在需求遗漏、误解、错做或需求外扩展。

## C. Plan Completion

逐条检查计划是否真正执行：

1. 计划中的步骤是否都有对应代码落点。
2. 是否存在跳步、漏步、顺序失真或计划状态失真。
3. 计划中的“已开发/开发中/未开发”是否与代码现状一致。
4. 已实现但未写入计划的内容，是否需要补充到计划。
5. 已写入计划但实际未实现的内容，是否需要回退状态或补充说明。

## D. Reuse And Consistency

这是强制检查项。

1. 是否优先复用项目内已有工具类、公共方法、公共组件、统一封装和相似业务实现。
2. 若未复用，是否存在明确且合理的原因。
3. 接口响应格式、错误处理方式、返回包装是否与项目现有接口风格一致。
4. DTO/VO 组织、分层方式、调用链结构是否与同类模块一致。
5. 页面交互模式、状态组织方式、数据流向是否与同类页面一致。
6. 是否引入了重复造轮子、风格断层或局部实现与全局约定冲突的问题。

## E. Comment Completeness

这是强制检查项。

1. 每个类是否有详细注释，说明职责、使用场景、上下游关系。
2. 每个字段是否有详细注释，说明业务语义、取值约束、特殊约定、单位、来源或去向。
3. 每个方法或函数是否有详细注释，说明用途、参数、返回值、前置条件、副作用、异常语义。
4. 核心业务逻辑代码块前是否有详细注释，说明业务意图、执行流程、关键判断、边界处理、兼容原因。
5. 注释是否真正解释了“为什么这样做”，而不是简单翻译代码。
6. 是否存在无信息量、模板化、敷衍式注释。

## F. Business Correctness

1. 状态流转是否正确。
2. 条件分支、计算规则、否决条件是否完整。
3. 空值、异常、失败返回、超时、重试、回滚是否处理合理。
4. 是否存在明显的业务回归风险。

## G. Performance And Maintainability

1. 是否存在重复查询、重复计算、无效 IO、无意义对象创建。
2. 是否存在明显可缓存、可批处理、可复用但未处理的点。
3. 函数、类、模块职责是否清晰。
4. 是否存在命名不清、结构混乱、过度耦合、难维护的问题。

## H. Verification Evidence

1. 本次实现是否完成了与改动匹配的最小充分验证。
2. 已执行的验证是否能证明需求条目和计划步骤已落地。
3. 是否明确记录了未验证项、未覆盖项和残留风险。

## I. Conclusion Rule

输出审查结论时，至少明确：

1. 当前实现是否符合需求文档。
2. 当前实现是否符合开发计划。
3. 哪些问题属于必须修复项。
4. 哪些问题属于可选优化项。
5. 开发计划是否需要同步更新。
