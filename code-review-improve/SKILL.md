# Code Review Improve

## Overview

这个技能不是围绕“当前会话改动”做泛化审查，而是围绕以下三类正式基线做代码审查：

1. 需求文档
2. 开发计划
3. 最近一次正式验证报告

目标是回答以下问题：

1. 需求文档中的验收标准是否真的已经被当前实现满足。
2. 开发计划中的前后端开发任务是否真的已经落地。
3. 当前代码是否与最近一次正式验证报告的结论一致。
4. 若存在问题，应该修代码、补验证，还是更新计划状态。

默认将 `references/validation-checklist.md` 作为统一审查清单使用。

## Trigger Guidance

以下表达应优先触发本技能或与本技能高度相关：

- “按需求和计划做代码审查”
- “结合测试报告看实现有没有问题”
- “看看现在代码有没有偏离验收标准”
- “按正式验证结果做一轮 review”
- “检查实现、验证报告和计划是否一致”

若用户只是让你随手看一段代码，没有需求文档、开发计划或正式验证基线，则不应把本技能当默认选择。

## Workflow

1. 锁定唯一需求文档、开发计划与最近一次正式验证报告。
2. 基于三类基线确定本次审查范围。
3. 审查验收标准达标情况、开发任务落地情况与验证结论一致性。
4. 审查业务逻辑、复用情况、风格一致性、注释完整性、性能与风险。
5. 对较大调整先与用户确认。
6. 进行最小必要修正，并在需要时更新开发计划或要求重新验证。
7. 按固定格式输出审查结论、问题清单、偏差说明与后续动作。

## Step 1: Lock Baseline

必须先锁定唯一基线，再进入正式审查。

优先根据名称或会话中的功能描述匹配：

- 需求文档：`docs/requirements/需求-{名称}.md`
- 开发计划：`docs/plans/开发计划-{名称}.md`
- 正式验证报告：`docs/check/tests/验证记录-{名称}-*.md`

规则：

1. 用户直接提供路径或内容时，以用户提供为准。
2. 若存在多个候选文档，先结合名称、内容、代码落点和最近时间做一次匹配判断。
3. 最近一次正式验证报告默认取同一验证对象下日期最新的一份。
4. 如果仍无法可靠锁定唯一基线，先向用户确认，不继续正式审查。
5. 一旦确认，后续所有结论都必须以该需求文档、开发计划和验证报告为基准。
6. 若需求文档与开发计划来自 `generate-requirements-and-plan`，其模板结构以该技能为唯一准绳：
   - 模板来源：`/Users/pengshuaifeng/.codex/skills/generate-requirements-and-plan/SKILL.md`
   - 本技能不重复定义整套模板，只围绕当前模板下的审查范围与审查结论展开。

## Step 2: Determine Review Scope

审查范围不是由“当前会话改动”决定，而是由需求文档、开发计划、正式验证报告和当前实现共同决定。

至少要识别：

1. 需求文档中的验收标准、页面要求、接口与数据要求。
2. 开发计划中的后端开发任务、前端开发任务、当前状态与任务进度。
3. 正式验证报告中的：
   - 验收标准验证矩阵
   - 开发任务验证矩阵
   - 未达标项
   - 阻塞项
   - 沿用历史结论项
4. 实际代码中的相关模块、类、方法、接口、页面、配置和数据结构。
5. 报告中已判定“达标”但当前代码可能已失效的部分。
6. 报告中未达标或阻塞、但代码可能已补齐的部分。

执行时先输出一句范围说明，例如：

- “本次按 `需求-XXX`、`开发计划-XXX` 与最新 `验证记录-XXX` 审查 `A/B/C` 模块，重点核对验收条目 `9.x` 与开发任务 `6.x/7.x` 的实际落地情况。”

## Step 3: Review Baseline Alignment

围绕三类基线逐项审查。

### 3.1 验收标准达标情况

逐条审查需求文档中的验收标准：

- 当前代码是否满足该条验收标准。
- 最新正式验证报告对该条的结论是否仍然成立。
- 若报告写“达标”，当前代码是否已经发生会使其失效的变化。
- 若报告写“部分达标 / 未达标 / 阻塞”，当前代码是否已经补齐或解除阻塞。

### 3.2 开发任务落地情况

逐项审查开发计划中的前后端开发任务：

- 当前代码是否真的覆盖该任务。
- 开发计划中的 `当前状态` 是否与实际实现一致。
- 最新正式验证报告对该任务的结论是否仍然成立。
- 是否存在“计划写已完成，但代码未落地”或“代码已完成，但计划未更新”的情况。

### 3.3 验证结论一致性

逐项审查正式验证报告与当前代码是否一致：

- 报告中的“达标”结论是否仍可信。
- 报告中的“沿用历史结论”是否仍满足沿用条件。
- 报告中的“未达标 / 阻塞”是否已经过时。
- 是否存在“代码已改，但报告未重测”的情况。

输出时应区分：

- 已一致项
- 报告已过期项
- 代码未达标项
- 计划状态失真项
- 需要重新验证项

## Step 4: Review Quality And Risks

在不脱离三类基线的前提下，按 `references/validation-checklist.md` 做审查。

重点检查：

1. 业务逻辑正确性
   - 状态流转、条件判断、计算规则、否决条件、兼容逻辑
2. 复用与一致性
   - 是否优先复用现有工具类、公共组件、统一封装、相似实现
   - 接口响应格式、异常处理、返回包装、DTO/VO 组织、页面交互模式是否与项目一致
3. 注释完整性
   - 类、字段、方法、核心业务逻辑是否有详细注释
4. 健壮性
   - 空值、异常、失败返回、超时、回滚、重试、边界输入
5. 性能与可维护性
   - 重复查询、重复计算、无效 IO、职责混乱、命名不清、过度耦合

原则：

1. 先找真实问题，再给最小改动建议。
2. 不为“看起来更优雅”而脱离计划大改。
3. 每个审查结论尽量映射到：
   - 验收标准
   - 开发任务
   - 验证报告条目
   - 具体代码位置

## Step 5: Confirm Before Large Changes

遇到以下情况先确认再改：

1. 业务逻辑上的较大调整。
2. 结构性重构或跨模块影响改动。
3. 需求文档、开发计划、验证报告与当前实现冲突且无法自动判定基准。
4. 修复一个问题需要顺带调整多处计划状态或使原验证结论失效。

沟通时必须说明：

1. 改什么。
2. 为什么改。
3. 对应哪条验收标准、哪个开发任务、哪份验证报告结论。
4. 影响哪些文件或模块。
5. 是否需要重新验证。

## Step 6: Apply Minimal Fixes And Follow-up Actions

若审查范围内的问题可以直接以最小改动修复，则进行修复；否则先给出明确建议。

后续动作按问题类型区分：

1. 代码问题：
   - 直接最小修复
2. 计划状态问题：
   - 更新开发计划中的任务状态、偏差说明或代码落点
3. 验证报告过期问题：
   - 明确标注“需要重新验证”
   - 必要时建议交由 `requirements-plan-test-verifier` 重新出具正式验证报告
4. 需求或计划本身冲突：
   - 明确列为待确认项，不擅自选边

若已匹配并确认开发计划文档，还应同步更新计划文档中的实现状态，避免“代码已变更但计划未更新”。

若用户未授权修改文档文件，则在输出中给出可直接应用的更新建议文本。

## Output Format

按固定顺序输出：

1. 基线信息
   - 需求文档路径
   - 开发计划路径
   - 正式验证报告路径
   - 本次审查范围
2. 审查结论概览
   - 验收标准一致性
   - 开发任务一致性
   - 验证报告一致性
   - 业务逻辑
   - 质量与风格
   - 性能与风险
3. 问题清单与建议
   - 按严重程度列出问题
   - 每项绑定验收标准、开发任务、验证报告条目和代码位置
4. 偏差与后续动作
   - 代码修复
   - 计划更新
   - 重新验证
   - 待确认项
5. 开发计划更新结果
   - 已更新或建议更新的内容
6. 验证处理建议
   - 哪些历史结论失效
   - 哪些条目必须复测

## Execution Rules

1. 没有锁定唯一需求文档、开发计划和正式验证报告前，不进入正式审查结论。
2. 审查范围默认由需求、计划和验证报告共同推导，不以“当前会话改动”作为默认基线。
3. 输出必须优先呈现问题和风险，再给摘要和建议。
4. 不确定的结论必须标注假设与待确认点。
5. 如果未发现明显问题，明确写“未发现明显问题”，并补充残余风险或测试盲区。
6. 若发现当前代码已使正式验证报告中的“达标”结论失效，必须明确标注“需要重新验证”。
7. 若发现开发计划状态与当前实现不一致，必须明确标注应更新的任务状态。
8. `references/validation-checklist.md` 是本技能的统一审查清单；审查和优化时应默认使用。
