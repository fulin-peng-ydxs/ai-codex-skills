# 自动化任务配置助手

当用户希望创建、更新、验证或排查 Codex 自动化任务时，使用此技能。

## 入口判断

- 如果用户要新建自动化或修改已有自动化，走“创建流程”。
- 如果用户说自动化失败、没有权限、没有运行、结果不对，或要求查看执行记录，走“诊断流程”。
- 当用户使用中文沟通时，自动化名称、提示词、状态汇报和最终结果都优先使用中文。

## 创建流程

1. 先简要说明两类自动化：
   - `cron`：按计划在一个或多个工作区独立运行。
   - `heartbeat`：在当前线程中稍后唤醒并继续。
2. 只收集缺失的必要输入：
   - 任务目标。
   - 运行时间、频率或触发条件。
   - 目标路径、仓库、URL、服务或线程上下文。
   - 需要的能力，例如文件写入、Git、网络、数据库、浏览器或凭据。
   - 结果汇报方式和失败处理方式。
3. 创建自动化前做本地预检查：
   - 检查目标路径是否存在。
   - Git 任务需要确认每个路径都是仓库，并查看 `git status`。
   - 检查引用到的技能、脚本、配置文件或凭据是否存在。
   - 识别可能需要授权的命令，尤其是带网络访问的 `git push`。
4. 使用 `codex_app.automation_update` 创建或更新自动化。
   - 周期性定时任务使用 `cron`。
   - 当前线程稍后继续使用 `heartbeat`。
   - 提示词必须自包含，写清楚路径、命令、成功标准和汇报格式。
   - 如果命令授权很重要，必须要求使用稳定的显式命令，例如 `git -C /path/to/repo push`。
   - Git 多仓库任务优先拆成多个自动化，每个自动化只绑定一个仓库 cwd；不要在一个自动化里配置多个 cwd 后又让提示词遍历多个仓库，否则手动执行或调度时可能出现重复运行和锁竞争。
5. 创建后做验证：
   - 查看自动化配置。
   - 本地运行安全的预检查命令。
   - 对网络或 push 工作流，实际触发一次命令授权，让后续运行能复用权限规则。
   - 尽量手动运行一次自动化本身，并检查 `~/.codex/automations/<id>/memory.md`。
   - 如果手动运行失败，必须根据运行记录修正自动化提示词、cwd 拆分或权限策略；只在当前线程验证同名命令不算完整验证。

## Codex 自动化网络处理

当 Codex cron 自动化需要访问网络，例如 `git push`、拉取远端、请求接口时，必须按下面流程处理：

1. 不要擅自把 Codex 自动化改成系统级 `launchd`、`cron` 或其他外部定时任务，除非用户明确要求换方案。
2. 先确认失败类型：
   - `Could not resolve hostname`、DNS 失败、network access 失败，通常表示自动化线程处在无网络沙箱。
   - 当前聊天线程里 `git push` 成功，不等于 Codex 自动化线程也有网络。
3. 对固定网络命令使用精确稳定命令，例如：
   - `git -C /Users/pengshuaifeng/.codex/skills push`
   - `git -C /Users/pengshuaifeng/.claude/skills push`
4. 在自动化提示词中写明：
   - 普通执行网络命令失败时，必须用 `require_escalated` 重新执行同一条命令。
   - 该命令需要匹配一个持久允许规则。
5. 持久允许规则涉及长期放开命令权限，必须先向用户说明风险并获得明确同意。
6. 用户同意后，才可在 `~/.codex/rules/default.rules` 中加入精确 `prefix_rule`。规则必须尽量窄，不要写宽泛的 `git push` 或 `git`。
7. 写入规则后，恢复或更新自动化，再手动运行自动化本身验证。验证通过后，后续定时运行才算闭环。

## 诊断流程

1. 定位自动化：
   - 如果用户没有提供自动化 ID 或名称，检查 `~/.codex/automations/*/automation.toml`。
   - 如果已知 ID，使用 `codex_app.automation_update` 的 `mode: view` 查看配置。
2. 查看运行记录：
   - 如果存在 `~/.codex/automations/<id>/memory.md`，优先读取。
   - 在 `~/.codex/logs_2.sqlite` 中搜索自动化 ID、名称、目标路径和错误关键字。
   - 如果 SQLite 日志上下文不足，再查看 `~/.codex/log/codex-tui.log`。
3. 归类失败原因：
   - 文件系统权限或沙箱写入限制。
   - Git 仓库状态问题。
   - 网络或 DNS 限制。
   - Git 远端凭据失败。
   - 缺少依赖、文件、技能或脚本。
   - 自动化提示词不够明确，导致选择的命令没有命中已授权规则。
   - 多个 cwd 触发多次运行，导致同一仓库并发提交或 `.git/index.lock` 冲突。
4. 修复可修复的问题：
   - 更新自动化提示词，改用稳定的显式命令。
   - 对需要网络或写入的命令申请授权。
   - 如果需要无人值守网络访问，向用户说明并申请写入精确持久 `prefix_rule`。
   - 对需要用户凭据、外部服务访问或手动 App 设置的阻塞点，明确说明。
