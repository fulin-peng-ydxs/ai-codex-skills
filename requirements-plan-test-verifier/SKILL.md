# Requirements Plan Test Verifier

## Overview

执行“锁定基线 -> 提取验收与任务 -> 建立双向映射 -> 生成验证 -> 执行验证 -> 输出正式报告”的闭环。

本技能是“需求文档 + 开发计划专属验证”技能，不再使用泛化补测定位。唯一验证基线只有两类：

- 需求文档中的验收标准
- 开发计划中的前后端开发任务

本技能的目标不是泛化扫描，而是回答以下问题：

1. 需求文档中的每条验收标准是否已经达标。
2. 开发计划中的每个前后端开发任务是否已经被验证覆盖。
3. 哪些验收标准或开发任务仍未达标、未覆盖或受阻塞。
4. 本轮验证结果能否作为后续复测的稳定基准。


## Workflow

1. 锁定唯一需求文档与开发计划。
2. 检查是否存在历史验证报告，并识别可沿用结论。
3. 提取需求文档验收标准与开发计划开发任务。
4. 建立“验收标准 <-> 开发任务 <-> 验证用例”双向映射。
5. 按后端和前端分层执行验证。
6. 输出可复测的正式验证报告。

## Step 1: 锁定唯一验证基线

必须先锁定唯一基线：

- 需求文档：`docs/requirements/需求-{名称}.md`
- 开发计划：`docs/plans/开发计划-{名称}.md`

规则：

1. 优先使用用户明确指定的文档路径或名称。
2. 若存在多个候选文档，先结合名称、内容、代码落点做唯一匹配。
3. 若无法可靠锁定唯一需求文档与开发计划，先向用户确认，不进入验证实现。
4. 若缺少需求文档或开发计划，不再按代码反推范围，也不降级成会话补测。
5. 若文档由 `generate-requirements-and-plan` 生成或维护，则其固定模板以该技能为唯一准绳：
   - 模板来源：`/Users/pengshuaifeng/.codex/skills/generate-requirements-and-plan/SKILL.md`
   - 本技能不重复定义整套需求文档与开发计划模板，只提取验证所需内容。

## Step 1.5: 检查历史验证报告

在开始建立本轮验证矩阵前，必须先检查是否存在同一验证对象的历史验证报告。

优先查找：

- `docs/check/tests/验证记录-{名称}-*.md`

若存在历史报告，至少提取：

- 上次报告的日期
- 上次报告使用的需求文档和开发计划
- 上次报告中的验收标准验证矩阵结论
- 上次报告中的开发任务验证矩阵结论
- 上次报告中的阻塞项、未达标项和复测建议

历史报告的作用：

1. 作为本轮复测基线。
2. 帮助识别哪些验收条目和开发任务可能可以沿用历史结论。
3. 帮助识别哪些条目必须重新验证。

但不得因为“历史报告显示通过”就直接跳过本轮验证。只有同时满足以下条件，才允许沿用历史结论而不重复执行：

1. 对应需求文档验收条目未发生变化。
2. 对应开发计划任务未发生变化。
3. 对应代码或页面落点自上次验证后未发生影响该条目的变更。
4. 上次结论不是“部分达标”“未达标”“阻塞”。
5. 当前轮次不存在新的相关缺陷、回归风险或依赖变更。

若不同时满足上述条件，则该条目必须重新验证。

## Step 2: 提取验收基线与任务基线

提取依据时，必须同时覆盖两条主线。

### 2.1 需求文档主线

至少提取：

- 页面需求明细中的页面、交互、字段、状态规则
- 接口与数据需求中的接口、字段、数据约束
- 非功能要求中的性能、安全、权限、审计、兼容要求
- 验收标准中的全部验收条目
- 待确认项中的未决约束和默认处理

其中：

- `验收标准` 是正式验证的首要基线。
- `待确认项` 会影响验证结论，必须体现在报告中，但不能替代验收标准本身。

### 2.2 开发计划主线

至少提取：

- 后端开发计划中的所有实际开发任务
- 前端开发计划中的所有实际开发任务
- 联调计划、测试计划、风险与回滚中的验证前置条件与阻塞信息
- 任务进度记录中的当前状态与历史推进情况

提取开发任务时，必须做到：

1. 后端任务按开发计划中的实际模块和任务粒度展开。
2. 前端任务按开发计划中的实际模块和任务粒度展开。
3. 不能只看章节标题，必须继续读取每个任务的目标、改动范围、复用锚点、一致性要求和当前状态。
4. 若某任务写明“无改动”或“未开始”，也必须进入验证矩阵并标记状态。

## Step 3: 建立双向验证映射矩阵

必须建立两张主矩阵，并保证双向可追踪。

### 3.1 验收标准验证矩阵

每条验收标准都必须映射到：

- 对应开发任务
- 对应验证方式
- 对应代码或页面落点
- 当前验证结果

推荐字段：

- 验收条目
- 对应需求章节
- 对应开发任务
- 验证方式
- 代码/页面落点
- 当前结果
- 证据
- 未达标或阻塞原因

### 3.2 开发任务验证矩阵

每个开发任务都必须映射到：

- 对应验收标准
- 对应验证方式
- 对应代码或页面落点
- 当前验证结果

推荐字段：

- 开发任务
- 对应计划章节
- 对应验收标准
- 验证方式
- 代码/页面落点
- 当前结果
- 证据
- 未达标或阻塞原因

### 3.3 映射规则

1. 不允许只建立“验收标准 -> 验证用例”单向映射。
2. 不允许只建立“开发任务 -> 验证用例”单向映射。
3. 需求文档中的每条验收标准都必须至少出现一次。
4. 开发计划中的每个开发任务都必须至少出现一次。
5. 若某条验收标准或某个开发任务暂时无法映射，必须明确写原因：
   - 当前代码未实现
   - 本轮环境阻塞
   - 依赖未完成
   - 文档信息不足
6. 验证结果统一使用：
   - 达标
   - 部分达标
   - 未达标
   - 阻塞
   - 不涉及
7. 若沿用历史报告结论，必须在矩阵中明确标注“沿用历史结论”，并记录对应历史报告路径或日期。
8. 若条目在本轮重新验证，也应明确标注“本轮复测”。

## Step 4: 执行验证

执行验证必须围绕映射矩阵展开，不允许脱离“验收标准 <-> 开发任务”自由补测。

整体要求：

1. 至少提供并执行一次“单命令批量运行”的方式。
2. 后端和前端分开组织验证内容、验证命令和验证结论。
3. 每项验证都要能回填到验收标准验证矩阵和开发任务验证矩阵。
4. 若受环境限制无法执行，必须明确记录未执行项、阻塞原因、替代验证和受影响条目。

### 4.1 后端执行验证

后端验证按开发计划中的后端开发任务逐项建立，至少包含以下两类：

#### 4.1.1 构建测试

目标：

- 证明后端代码可编译、可打包或至少可完成最小构建检查。

可选命令按项目实际选择：

- `mvn test`
- `mvn -pl <module> -am test`
- `mvn -DskipTests compile`
- 其他项目内可执行测试或构建命令

要求：

1. 至少有一条能证明当前后端改动未破坏编译或测试加载的命令。
2. 若只做编译不做测试，必须说明原因，且不能将相关验收条目标记为“达标”。

#### 4.1.2 单元测试

后端单元测试必须拆成两个动作：

1. 生成测试用例
2. 执行测试用例

生成测试用例时：

- 优先复用现有测试基类、测试配置、数据初始化方式。
- 优先覆盖 Service 层或核心业务逻辑。
- 必要时补充 Controller 切片测试或集成测试。
- 使用 Mock 或测试桩隔离 Feign、MQTT、Redis、第三方服务等外部依赖。
- 每个测试用例都要明确覆盖哪些验收标准和哪些后端开发任务。

执行测试用例时：

- 必须执行本轮新增或修改的测试。
- 测试命名、断言风格、目录结构要与项目现有测试保持一致。
- 不能出现“生成了测试但未执行”的达标结论。

### 4.2 前端执行验证

若开发计划中存在前端开发任务，默认必须补前端验证，不允许只做后端单测。

前端验证建议至少按以下层次组织：

#### 4.2.1 构建测试

目标：

- 证明前端代码在当前工程中可通过最小构建或类型检查。

可选命令按项目实际选择：

- `pnpm build`
- `pnpm exec vue-tsc --noEmit`
- `pnpm lint`

要求：

1. 至少执行一条能证明前端构建、类型或静态检查不过早失败的命令。
2. 若只做静态检查未做交互验证，不能将涉及页面行为的验收条目标记为“完全达标”。

#### 4.2.2 页面与组件交互验证

目标：

- 验证页面、组件、弹窗、表单、按钮和联动行为是否符合需求文档与前端开发任务。

可执行方式按项目基础选择：

- 组件测试
- 页面级测试
- 最小挂载验证
- 人机脚本式自动化验证

重点覆盖：

- 查询字段
- 展示字段
- 表单和弹窗字段
- 默认值
- 排序、筛选、分页
- 高亮、颜色、显隐、禁用
- 页面跳转、按钮点击、提交和重置行为

#### 4.2.3 接口契约与字段映射校验

目标：

- 验证前端与后端接口在参数、字段、状态和值语义上的一致性。

至少检查：

- 请求参数是否一致
- 返回字段是否一致
- 空值语义是否一致
- 状态字段和枚举字段是否一致
- 错误提示和异常处理是否一致

#### 4.2.4 关键流程烟雾验证

目标：

- 验证最关键的一条页面主流程能走通。

可执行方式：

- Playwright / Cypress / Vitest + 挂载测试
- 若无现成 E2E 能力，可使用最小流程级页面验证替代

要求：

1. 至少覆盖一条主流程。
2. 若涉及异常或边界高风险点，至少补一条失败或边界链路。

### 4.3 前后端联动验证

若验收标准或开发任务涉及前后端联动，必须至少验证：

- 请求参数是否一致
- 返回字段是否一致
- 空值语义是否一致
- 状态字段和枚举字段是否一致
- 成功链路和至少一条失败或边界链路

### 4.4 执行结论规则

1. 后端新增或修改测试必须能够执行。
2. 前端自动化验证必须尽量执行，不得只写“建议验证”。
3. 不能仅凭构建成功就判定页面交互类验收已达标。
4. 不能仅凭单元测试通过就判定前端任务已达标。
5. 若某项只能做部分验证，矩阵结果应标记为“部分达标”，不能直接写“达标”。

## Step 5: 输出正式验证报告

在 `docs/check/tests` 下新增或更新报告，文件名建议：

- `验证记录-{名称}-{YYYYMMDD}.md`

若已有同名历史报告：

1. 优先参考最近一次报告中的矩阵结构与验证结论。
2. 本轮报告应明确哪些条目沿用了历史结论，哪些条目进行了本轮复测。
3. 本轮报告应明确哪些条目结论发生变化。
4. 不覆盖历史报告，保留可追溯的复测基线。

报告结构使用 [test-report-template.md](references/test-report-template.md)，至少必须包含：

- 基线文档信息
- 验收标准验证矩阵
- 开发任务验证矩阵
- 本次新增或修改的测试与验证落点
- 后端与前端执行命令及结果
- 未达标项、阻塞项与原因
- 本轮结论和下轮复测基线说明

## Execution Rules

- 没有唯一需求文档与开发计划，不进入正式验证。
- 正式验证只能以“需求文档验收标准 + 开发计划开发任务”为基线，不按代码反推范围替代。
- 每条验收标准都必须有验证结论；每个开发任务都必须有验证结论。
- 报告中必须同时保留“验收标准验证矩阵”和“开发任务验证矩阵”，二者不得缺一。
- 测试实现必须与映射矩阵对应，不能出现“有测试但不知道覆盖了什么”。
- 允许沿用历史报告中的“已达标”结论，但必须先完成变更影响判断，不能无条件跳过复测。
- 若当前计划涉及前端开发任务，不得只做后端单测就宣称验证通过。
- 不得伪造测试执行结果；未执行成功时必须说明阻塞原因。
- 本技能默认输出正式验证报告，不默认回填开发计划；若用户明确要求，再单独更新计划文档。
