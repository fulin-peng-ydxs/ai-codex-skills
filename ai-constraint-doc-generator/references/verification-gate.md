# 验证门禁

## 门禁原则

只要某条命令会作为“必须执行的日常流程”写入 `AGENTS.md` 或 `CLAUDE.md`，它就必须在本次任务中实际成功执行。非零退出、缺少工具、健康检查失败或超时都视为门禁失败。

门禁失败时，可以在对话中汇报发现和未落地草案，但不能覆盖目标说明文件，直到用户选择处理方式并且必要门禁通过。

## 必备验证计划

编辑文件前先形成验证计划，至少包含：

- `stack`：检测到的语言、框架或组件。
- `source`：证明命令相关的清单文件、脚本、README 章节或已有说明文件。
- `command`：精确命令和工作目录。
- `purpose`：环境、依赖、测试、构建、本地启动、健康检查、状态检查、停止清理或领域 CLI。
- `required_for_landing`：是否属于落地必验命令。
- `success_condition`：退出码、预期输出、监听端口、HTTP 状态、生成产物或测试结果。
- `cleanup`：若命令会启动服务，写清停止命令或进程清理方式。

遇到破坏性、生产环境、耗时昂贵、需要凭证、写外部服务或可能长时间运行的命令，先询问用户。

## 必须通过的内容

至少验证所有将写入文档的常规工作流命令：

- 环境：运行时版本和包管理器版本。
- 依赖：仓库安全允许的安装或依赖可用性检查。
- 测试：每个检测到的技术栈的主测试命令，除非项目没有测试或用户明确只要求命令发现。
- 构建：每个可构建技术栈的生产或打包构建命令。
- 本地启动：文档会写启动流程时，验证启动、状态或健康检查、停止命令。

如果仓库提供封装脚本，优先验证封装脚本接口，而不是裸跑底层长期命令。

## 长期服务

验证本地启动命令时：

1. 有状态命令时先检查当前状态。
2. 仓库有自动化专用端口时，不抢占用户日常开发端口。
3. 服务只启动到足够验证就绪。
4. 用状态命令、监听端口、HTTP 响应或健康检查验证就绪。
5. 只停止本次验证启动的进程；仓库生命周期规则另有要求时按仓库规则恢复。
6. 仓库要求恢复测试前状态时，按测试前状态恢复。

## 失败处理

命令失败时：

1. 记录命令、工作目录、退出状态和最短有效错误片段。
2. 将失败归类为缺少依赖、运行时版本不匹配、测试失败、构建失败、端口冲突、外部服务缺失、密钥缺失、数据锁、超时或未知。
3. 只有存在明确、非破坏性修复依据时，才做一次定向重试，例如安装已声明依赖、改用仓库封装脚本、设置明确文档化的本地环境文件、清理由本 Agent 启动的进程。
4. 重试失败或修复方式不确定时，停止并询问用户如何处理。
5. 不在没有新证据的情况下连续尝试命令变体。

门禁失败时不落地说明文件。用户需要阶段性结果时，可在对话中给未落地草案；只有用户明确允许时，才写入带草案标识的独立文件。

## 汇报证据

成功落地后汇报：

- 写入的文件。
- 按技术栈分组的已验证命令。
- 未写入命令及原因：不存在、不相关、不安全或用户未要求。
- 残余风险，例如本地检查未覆盖的外部服务。
