# 需求文档生成器

## 核心目标

将用户的初步功能想法整理成一份能指导后续 AI 制定高质量开发计划的 `requirement.md`。优先依据当前项目的真实代码、数据库、业务流程和已有交互模式，不要只输出通用产品描述。

## 执行流程

1. 确定功能名称和输出目录。
   - 输出目录必须相对目标项目根目录，而不是无关的空对话目录或技能管理目录。
   - 如果当前工作区明显不是目标项目，或没有可用于调研的业务代码但用户要求结合项目现状，先确认目标项目路径，不要直接写入当前目录。
   - 如果用户指定了目录，使用用户指定目录。
   - 如果用户未指定目录，根据功能名生成精确 slug，输出到 `./agent-works/{feature-slug}/requirement.md`。
   - 生成前检查目标 `requirement.md` 是否已存在；若已存在且用户没有明确要求更新该文件，不复用、不覆盖，改用更精细的 `{feature-slug}`。
   - 细化 `{feature-slug}` 时优先使用稳定业务语义，例如业务对象、流程、动作、页面、场景或配置范围；避免只用日期、随机数、`new`、`v2` 这类弱语义后缀。

2. 先判断是否已有上游“需求闭环分析”结果。
   - 如果用户已经通过 `requirement-closure-designer` 或其他方式确认过“需求闭环分析”“完整功能闭环设计稿”“页面闭环草案”或同类上游结果，先把这些内容视为正式需求文档的主要输入，不要重新发散生成另一套需求闭环。
   - 对上游已确认内容，默认按“已确认事实”进入正式需求文档；只有当项目现状、代码实现、数据库结构、权限模型或用户新增约束与之冲突时，才指出冲突并重新确认。
   - 对上游输出中的“阻塞确认项”“非阻塞确认项”“推断决策”“必须实现 / 建议实现 / 暂不实现”范围划分和建议项处理状态，要优先继承和映射，不要在无冲突前提下改写含义。
   - 对上游已经明确的页面入口、页面职责、关键操作链路、状态流转、数据变化、权限边界和异常反馈，在正式需求文档中必须保留并展开，不得遗漏后重新生成一套页面承载关系。
   - 如果用户只给了闭环分析结果但没有明确要求生成正式需求文档，先确认是否进入 `requirement.md` 生成，不要自动落文档。

3. 先调研项目现状，再写需求。
   - 阅读相关模块、页面、接口、服务、权限、配置和测试。
   - 如当前环境存在数据库 MCP 或可用数据库信息，结合表结构和实际数据分析需求。
   - 在文档中记录具体依据，例如文件路径、接口名、表名、字段名、枚举值、已有业务规则。
   - 如果无法访问数据库或关键上下文，要在文档中明确说明限制，不要编造事实。

4. 采用专业角色视角补全需求。
   - 执行本技能时，默认以“资深业务领域专家 + 顶级架构师级别开发者”的双重视角审视需求。
   - 业务领域不作为文档章节输出，只用于判断需求是否专业、业务闭环是否完整、术语是否准确、规则是否符合行业或项目语境、用户交互是否贴合真实使用场景。
   - 如果用户明确了领域，按用户给出的领域分析；如果未明确，结合初步需求、当前项目模块、已有业务流程、数据库表、页面交互和项目命名习惯自然推断。
   - 如果领域无法稳定推断，不要强行编造具体行业，继续以“当前项目业务专家 + 顶级架构师级别开发者”的视角分析。
   - 只有当领域差异会明显改变需求边界、业务规则、专业术语、合规要求或用户体验，且项目现状无法判断时，才向用户确认具体领域。

5. 识别前端页面、交互和配置化需求。
   - 只要需求涉及新页面、更新页面、页面布局、组件、弹窗/抽屉、表格、图表、表单、设置项、偏好记忆、配置中心或通用前端能力，就必须先检查项目是否已有设计规范、组件库、页面模板、布局约定、样板页面或视觉验收方式；有则作为项目依据写入文档，没有则写明本次采用的最低交互约束。
   - 不要假设项目一定已有设计系统、布局系统或配置平台；先判断这是普通页面、复杂工作台、可配置页面，还是需要建设可复用基础能力。
   - “页面与交互开发需求”必须引用对应需求编号，并按页面和页面功能点组织；页面功能点必须覆盖本次需求范围内所有用户可触发、系统可反馈、状态会变化、数据会展示变化或提交变化的交互。
   - 每个页面条目必须说明页面结构、功能关系和变更范围；需要图示时，将页面布局图、功能关系图或变更区域图放到对应页面条目中，不要单独堆一段图示规则。
   - 如果需求包含新增页面，必须为每个新增页面生成对应 HTML 原型文件，并在页面条目中引用原型路径；原型文件放在需求文档输出目录的 `prototypes/` 子目录下，命名使用稳定页面语义，例如 `prototypes/{page-slug}.html`。
   - HTML 原型必须优先参考项目已有前端风格、组件库、布局密度、导航结构、表格/表单/弹窗模式和视觉规范；如果项目缺少可参考风格，原型应采用克制、可实现、便于开发理解的低保真布局，不要把营销页、装饰性视觉或无关交互动效写入业务原型。
   - HTML 原型中的功能交互必须与需求定义的实际功能一致；出现按钮、菜单、筛选、提交、弹窗、抽屉、切换、联动、状态变化等交互元素时，必须提供对应的可用交互反馈或明确标注为禁用/暂不实现，不能放置看似可用但无行为的功能控件。
   - HTML 原型中用于解释业务规则、字段含义、流程原因、验收口径或开发注意事项的说明内容，必须用区别于正式业务界面的颜色、边框、背景或标签样式标识为“原型说明/开发说明”，并避免伪装成真实页面文案、提示条、表单字段或业务组件，防止后续开发把说明性内容误做成线上功能。
   - 如果只是更新既有页面，不要一律生成原型；当变更涉及新增主要区域、页面布局重排、跨组件联动、复杂表单/表格/图表、工作台、配置页、可视化状态或会明显影响开发理解的交互时，应生成更新后原型或变更区域原型。仅涉及文案、字段、小按钮、简单显隐、局部校验或低风险状态提示时，用页面布局说明、变更区域说明或图示即可，不强制生成 HTML 原型。
   - 如果需求包含“可配置”“自定义”“偏好”“模板”“预设”等能力，必须说明谁配置、配置什么、配置保存在哪里、如何生效、如何失效或回退、默认值是什么、非法值如何处理。
   - 任何配置项都必须说明展示入口、保存位置、读取位置、实际消费位置、默认值、非法值处理和验收方式；不能只要求“保存配置”而不说明谁消费。
   - 当需求明显会被多个页面、模块或场景复用时，必须把它识别为基础能力建设，补充样板场景、渐进接入范围和不做范围，避免把通用能力按单个页面功能草率定义。

6. 对不确定事项先分类，再决定是否确认。
   - 阻塞确认：缺失答案会导致需求文档无法闭环，或会直接改变业务流程、数据结构、权限范围、状态流转、集成契约、开发方案、验收标准。
   - 非阻塞确认：缺失答案不影响开发计划启动，但会影响交互细节、默认值、提示文案、次要操作、展示顺序或实现复杂度。
   - 自行推断：项目已有同类模式、低风险、可回滚、不会改变数据结构或业务闭环的细节。
   - 对每个不确定项给出分类理由和影响，不要只罗列问题。
   - 阻塞确认必须先问用户。
   - 非阻塞确认先列给用户，并询问是否需要一并确认；如果用户选择确认，将这些问题升级为阻塞确认并逐项确认；如果用户选择不确认，则写入“遗留确认项”或按项目惯例形成“推断决策”。
   - 自行推断项直接写入文档并标注依据。
   - 如果待确认问题很多，先给出完整分类清单，再从最高影响的问题开始确认。不要设置固定数量上限，但每批问题要保持用户容易回答。
   - 如果上游闭环分析中已有同名确认项或推断决策，优先沿用原结论；只有在项目现状或用户新增信息导致结论失效时，才升级为新的确认问题。

7. 不要过早生成最终文档。
   - 阻塞确认问题没有解决前，不要直接写最终 `requirement.md`。
   - 如果用户明确要求先按假设推进，要在文档中标注“推断决策”。
   - 当需求涉及页面开发时，在写入最终 `requirement.md` 和必要原型文件前，必须先向用户展示待落地内容摘要，包括输出目录、页面清单、哪些页面需要 HTML 原型、哪些更新页面不需要原型及原因、关键布局假设、仍需确认的问题；只有用户明确确认后，才创建或覆盖 `requirement.md` 与必要的 `prototypes/*.html`。
   - 如果用户要求先产出草案，不要把草案当作最终落地文档；应先在对话中给出草案摘要或待确认清单，待确认后再写入文件。

8. 写入最终需求文档。
   - 必要时创建输出目录。
   - 如果存在需要 HTML 原型的页面，先创建或更新输出目录下的 `prototypes/` 子目录和对应原型文件，再在 `requirement.md` 页面条目中引用相对路径，例如 `prototypes/{page-slug}.html`。
   - 文档要足够具体，能直接指导后续开发计划和测试设计。
   - 为每个可追踪需求项分配稳定需求编号，便于后续开发计划建立需求与开发内容映射关系。
   - 区分已确认事实、项目依据、推断决策和风险。
   - “建议实现”和“业务亮点与优化建议”中的可执行条目必须带处理状态，便于后续判断是否已纳入、已实现或暂不处理。
   - 如果仍有未解决但不阻塞开发计划的问题，在文档末尾增加“遗留确认项”；如果关键问题未确认，不要生成最终文档，先继续确认。
   - 如果存在上游闭环分析结果，正式需求文档应以“继承并展开”作为默认策略，而不是“重新构造一套平行需求”。正式文档中的功能范围、详细功能需求、页面与交互开发需求、验收标准，应能追溯到上游闭环结果中的功能点、页面闭环草案、范围划分和确认结论。
   - 始终保留“建议开发顺序”章节；需求规模很小时写明建议单阶段完成，需求较大、跨多个模块或后续可能继续拆分时，给出基于本需求文档的模块拆分和推荐推进顺序。

## 需求编号规则

- 使用 `R1`、`R2` 这类本需求文档内稳定编号标识需求项；同一需求项在功能范围、详细功能需求、数据与接口需求、页面与交互开发需求和验收标准中必须使用同一编号。
- “必须实现”的需求项必须编号；“建议实现”和“暂不实现”也应编号，便于开发计划明确纳入、排除或后续处理。
- “建议实现”的条目必须带处理状态，状态只使用：`本次纳入`、`后续建议`、`已实现`、`待确认`、`不纳入`。`已实现` 必须有项目现状或后续实际落地依据。
- 一个编号只表达一个可验收的需求项；不要把多个无依赖关系的能力合并到同一个编号。
- 编号不得复用或在文档内改变含义；需求变更时新增编号或明确标注原编号的范围调整。
- 每个编号都要能追溯到用户诉求、项目依据、推断决策或确认结论；不能把未确认想法包装成已确认需求。

## 质量要求

- 尽量用项目代码和数据库证据支撑判断。
- 明确标注未知信息，不要把猜测写成事实。
- 优先沿用当前项目已有模式，不轻易引入新抽象。
- 如果已有上游闭环分析结果，不要把它丢掉后重新生成一套需求闭环；应在保留原闭环语义的基础上补全为正式需求文档。
- 根据需求相关性覆盖增删改查、查询筛选、导入导出、状态流转、权限边界、数据一致性、空状态、错误状态、加载状态、并发、大数据量和兼容性。
- 验收标准必须可观察、可测试。
- 需求编号必须足以支撑后续开发计划的“需求与开发内容映射和开发状态”章节；不能只在功能范围中编号，却在详细需求和验收标准中丢失编号。
- “建议实现”和“业务亮点与优化建议”中的可执行条目必须有处理状态；后续更新需求文档时，要按真实开发结果同步状态，不能只保留建议文字。
- 涉及前端页面时，不能只有泛泛文字说明，必须在具体页面条目的“页面布局”中表达页面区域、主次关系、核心模块、操作路径、数据区、弹窗/抽屉/设置入口、关键状态或退化方式，并按需求编号列清页面、页面功能点和对应交互。
- 涉及新增页面时，需求文档不能只有文字布局说明，必须引用同目录 `prototypes/` 下的 HTML 原型文件；更新页面按复杂度决定是否生成原型，并在页面条目中说明判断理由。
- HTML 原型必须服务于需求澄清和后续开发理解，优先复用项目既有视觉、布局和组件模式；不得脱离项目风格生成难以落地的展示稿。
- HTML 原型必须避免无效交互控件；所有看起来可点击、可输入、可切换或会触发数据变化的元素，都要与需求中的真实功能、状态反馈和验收标准一致，无法实现或暂不纳入的交互必须禁用或显著标注。
- HTML 原型里的业务说明、开发说明和验收说明必须有统一的视觉区分，不得混入真实页面内容层级，避免开发阶段误把说明文字实现为业务功能。
- 涉及页面开发的最终文档和原型文件必须经过用户确认后再写入，不能在阻塞确认未完成或用户未确认落地前直接创建最终 `requirement.md`。
- 页面与交互开发需求必须覆盖本次需求范围内所有用户可触发、系统可反馈、状态会变化、数据会展示变化或提交变化的交互；不能只写页面布局或核心按钮。
- 如果上游闭环分析已经明确了页面入口、页面职责、关键操作链路、状态流转、数据变化、权限边界和异常反馈，正式需求文档必须保留这些页面承载关系，不得遗漏后另起一套页面设计逻辑。
- “建议开发顺序”必须基于需求依赖、模块耦合、基础能力前置关系、页面承载关系和可验证里程碑来排序，不能随意按想到什么写什么，也不能直接退化成开发任务清单。
- 当需求较大、模块较多或页面/能力之间依赖明显时，必须让“建议开发顺序”具备后续继续拆出专属需求文档或开发计划的指引价值。
- 涉及配置化、可定制、偏好记忆或模板能力时，必须写清“谁定义、谁渲染、谁保存、谁读取、谁消费、谁校验、如何回退”。
