# Development Trace

为当前会话中的业务文件改动生成可追溯的开发记录，沉淀“为什么做、实际做成什么、具体怎么做”。

## 适用范围

在以下场景使用本技能：

- 用户要求对当前会话改动、当前 Git 改动或指定文件做开发留痕。
- 用户要求根据开发计划、需求文档、实现代码生成 `agent-works` 下的说明文档。
- 用户要求按功能需求拆分总结，而不是只按文件列表罗列改动。

本技能只生成或更新文档，不修改业务代码。

本技能只沉淀开发过程、实现结果和证据来源，不替代业务功能核查；如果用户要求判断业务链路是否成立、状态流转是否闭环或上下游协同是否完整，应使用或建议使用 `business-feature-audit`。

## 证据来源与默认范围

按以下顺序收集证据，结论必须能从证据中推导：

1. 用户在当前会话提出的功能需求、补充说明和调整意见。
2. 当前会话中实际新增或修改的业务文件。
3. Git 工作区改动：`git status --short`、`git diff --stat`、必要时查看具体 `git diff -- <file>`。
4. 项目已有需求和开发计划：优先查找项目内 `./agent-works/`、用户指定需求文档、当前会话中明确引用的计划或文档；如果当前会话或用户明确给出 Codex 开发计划 slug，可读取 `~/.codex/plans/{plan-slug}.md`。
5. 与改动直接相关的代码上下文：控制器、服务、Mapper、前端页面、组件、接口封装、路由、配置等。

默认只把当前会话明确涉及、且本次可从代码或 diff 验证的业务文件纳入留痕范围；即使 Git 工作区存在其他未提交改动，也不得默认写入本次留痕。只有用户明确要求“处理所有改动”“基于当前 Git 工作区全部改动生成留痕”“包括所有未提交文件”等含义时，才可把整个工作区改动纳入范围。

仍必须查看项目内 `./agent-works/`、需求文档、开发计划、历史 `development-trace.md` 和 `business-audit.md`，用于复用功能目录、确认功能边界和补充历史上下文。

不要把未查看、未验证、无法从当前证据支撑的内容写成事实。不得全局搜索 `~/.codex` 中未明确关联的历史会话、缓存或自动化日志作为事实依据；读取 `~/.codex/plans/{plan-slug}.md` 前，必须已经从用户输入或当前会话得到明确的 `plan-slug`，不得为了寻找计划而枚举整个 `~/.codex/plans/`。

## 功能拆分与计划复用

以“功能需求”为单位拆分文档，不以文件数量拆分。

- 多个文件共同完成同一个业务能力时，合并成一个功能文档。
- 一个文件同时承载多个独立功能时，按功能拆成多个文档，并在每个文档中只写与该功能相关的实现。
- 纯格式化、无业务含义的改动不单独成文；如必须提及，放在相关功能的“实际结果”中。
- 如果功能边界不清，先根据用户需求、开发计划标题、页面/接口职责和 diff 内容归纳；仍无法判断时，向用户确认。

如果本功能之前生成过开发计划，必须先对照当前实际改动：

- 计划与实际一致：可以复用计划中的需求描述、任务拆分和方案，但要结合实际代码补全结果。
- 计划与实际有偏差：以原计划为基础说明最终实现，不能照搬过期内容。
- 计划只覆盖部分内容：复用可确认部分，其余根据会话和代码补充。
- 找不到计划：根据当前会话需求和代码改动重新归纳。

文档中不要写“计划已验证”“本次已测试”等未实际执行过的结论。

## 输出位置与命名

统一输出到项目根目录下的 `./agent-works/` 归档目录；文档属于具体需求、功能或模块时，优先放到：

```text
./agent-works/{feature-slug}/development-trace.md
```

目录与命名规则：

- `./agent-works/{feature-slug}/` 中的 `{feature-slug}` 是业务英文名或稳定功能英文名，使用小写 kebab-case。
- 同一功能的需求文档、开发计划、开发留痕、业务核查、SQL、测试记录、复盘记录等必须复用同一目录。
- 若 `./agent-works/` 已有相关分类或功能目录，优先复用；不要在项目根目录生成零散 Markdown，除非用户明确指定。
- 文档标题使用中文功能需求名称。
- 如果目标 `development-trace.md` 已存在，优先增量更新；保留仍然准确的历史内容，仅在内容明显错误、过期或与当前实现冲突时才删除或改写。

slug 规则：

1. 优先沿用已有 `agent-works` 目录、开发计划、业务核查文档或历史 `development-trace.md` 中已经使用的功能目录名。
2. 若无既有目录，根据中文功能需求生成稳定英文 slug；无法稳定翻译时，先在留痕方案中给出建议 slug，等待用户确认。
3. 同一功能需求不得反复生成不同 slug；如果本次改动属于已有功能的延续，应更新原目录而不是新建目录。
4. 不要使用过泛目录名，例如 `development-trace`、`feature-update`、`business-change`、`code-change`。
5. 一个需求拆成多个独立功能时，每个功能使用独立 slug；多个文件共同完成同一业务能力时，共用一个 slug。

必要时先创建 `agent-works` 及对应功能目录。

## 确认流程

写入或更新 `agent-works` 文档前，必须先向用户确认留痕方案。确认内容必须包含：

- 本次纳入留痕的文件范围：列出业务文件，并说明排除的构建产物、临时文件或无关文件。
- 默认范围说明：若用户未要求处理所有 Git 改动，说明本次仅纳入当前会话明确涉及且可验证的相关代码文件；其他工作区改动不纳入。
- 功能需求拆分结果：每个功能的中文标题、英文目录名 `{feature-slug}`、对应文件和拆分理由。
- 开发计划复用情况：列出找到的相关计划或说明未找到；说明每个功能准备复用、修正或重新归纳的依据。
- 上线部署配套项：说明本次是否涉及 SQL、配置、环境变量、脚本、字典、菜单权限、定时任务、第三方配置等非代码配套产物。
- 输出动作：标明是新建文档还是更新已有文档，以及目标路径。

确认规则：

1. 先展示“拟生成/更新的开发留痕方案”。
2. 明确等待用户确认；用户未确认前不得创建或修改 `agent-works` 文档。
3. 如果用户要求调整文件范围、功能拆分、标题、目录名或目标路径，先按反馈更新方案，再次确认。
4. 只有用户明确表示确认、同意、继续、可以生成等含义后，才执行写入。
5. 即使用户在最初请求中已经明确指定文件范围、功能拆分和输出文件名，写入前也仍需做一次最终确认。

## 文档模板

每个文档使用以下结构：

```markdown
# 功能需求

## 功能的详细需求

- 写清楚业务目标、使用场景、输入输出、边界条件和用户可见行为。
- 如果需求来自开发计划或会话补充，融合成当前最终口径。

## 功能开发的实际结果

- 写清楚当前代码实际完成了什么。
- 列出关键业务文件及各自承担的职责。
- 如有未完成、暂未覆盖或依赖外部条件的部分，明确说明。

## 功能开发的具体实施方案

- 按实现链路描述具体怎么做：前端入口、状态/表单/交互、接口调用、后端接口、服务逻辑、数据查询、异常处理、配置等。
- 只写与本功能有关的实现细节，避免泛泛描述。
- 如果复用了原开发计划，说明最终实现与计划是否一致，以及差异点。

## 上线部署操作

- 列出本功能上线时需要配套执行或确认的非代码产物，例如 SQL 文件、配置文件、环境变量、脚本、字典、菜单权限、定时任务、第三方平台配置等。
- 对每个配套项写明文件或配置位置、用途、执行或生效方式、目标环境，以及是否必须上线前执行。
- 如果本功能不涉及上线部署配套操作，明确写“本次无额外上线部署操作”。

## 验证情况

- 列出本次确实执行过的验证命令、脚本或检查结果。
- 如果未执行验证，说明原因；如果验证失败或存在阻塞，说明失败点和影响范围。
- 不得把未执行的验证写成已完成。
```

可以根据需要追加“关联文件”“注意事项”等章节，但“功能的详细需求”“功能开发的实际结果”“功能开发的具体实施方案”“上线部署操作”“验证情况”五个章节必须保留且内容完整。

## 执行流程

1. 定位项目根目录：优先使用 `git rev-parse --show-toplevel`；如果不是 Git 仓库，则以用户指定目录或当前工作目录为项目根。
2. 查看 `git status --short` 获取工作区背景范围；如果不是 Git 仓库，则根据用户指定文件和会话上下文确定背景范围。
3. 默认只识别当前会话明确涉及且可验证的相关代码文件作为本次留痕范围；只有用户明确要求处理所有改动时，才把完整 Git 工作区改动纳入本次范围。排除构建产物、依赖目录、临时文件和无关文档。
4. 查找相关需求文档、开发计划和已有 `agent-works` 文档。
5. 阅读必要 diff 和代码上下文，按功能需求分组。
6. 形成拟生成/更新方案，并按“确认流程”向用户确认。
7. 用户确认后，为每个功能生成或更新 `agent-works/{feature-slug}/development-trace.md`。
8. 完成后汇报项目根目录、生成或更新的文档路径、覆盖的功能需求、主要证据来源。

## 输出约束

- 文档内容使用中文。
- 未经用户确认留痕方案，不得创建或修改 `agent-works` 文档。
- 不复制敏感配置值；涉及配置时只写文件位置、用途和环境差异。
- 涉及 SQL、配置、环境变量、脚本、字典、菜单权限、定时任务、第三方平台配置等上线配套项时，必须在“上线部署操作”中列明；不能确认是否需要执行时，标记为待确认。
- 不夸大结果，不把推测写成事实。
- 不把验证写成已完成，除非本次确实执行了对应命令或脚本。
- 不为了留痕修改业务代码。
