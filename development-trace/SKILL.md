# Development Trace

为当前会话中的业务文件改动生成可追溯的开发记录。目标是把“为什么做、实际做成什么、具体怎么做”沉淀成面向项目交接和复盘的 Markdown 文档。

## 适用范围

在以下场景使用本技能：

- 用户要求对当前会话改动、当前 Git 改动或指定文件做开发留痕。
- 用户要求根据开发计划、需求文档、实现代码生成 `agent-works` 下的说明文档。
- 用户要求按功能需求拆分总结，而不是只按文件列表罗列改动。

本技能只生成或更新文档，不修改业务代码。

## 信息来源

按以下顺序收集证据，结论必须能从证据中推导：

1. 用户在当前会话提出的功能需求、补充说明和调整意见。
2. 当前会话中实际新增或修改的业务文件。
3. Git 工作区改动：`git status --short`、`git diff --stat`、必要时查看具体 `git diff -- <file>`。
4. 项目已有需求和开发计划：外部查找 `./agent-works/`目录，内部查找你自己的工作目录(~/.codex)下的可能相关的。
5. 代码上下文：控制器、服务、Mapper、前端页面、组件、接口封装、路由、配置等与改动直接相关的文件。

不要把未查看、未验证、无法从当前证据支撑的内容写成事实。

## 功能拆分

以“功能需求”为单位拆分文档，不以文件数量拆分。

拆分规则：

- 多个文件共同完成同一个业务能力时，合并成一个功能文档。
- 一个文件同时承载多个独立功能时，按功能拆成多个文档，并在每个文档中只写与该功能相关的实现。
- 纯格式化、无业务含义的改动不单独成文；如必须提及，放在相关功能的“实际结果”中。
- 如果功能边界不清，先根据用户需求、开发计划标题、页面/接口职责和 diff 内容归纳；仍无法判断时，向用户确认。

## 开发计划复用

如果本功能之前生成过开发计划，必须先对照当前实际改动：

- 计划与实际一致：可以复用计划中的需求描述、任务拆分和方案，但要结合实际代码补全结果。
- 计划与实际有偏差：以原计划为基础说明最终实现，不能照搬过期内容。
- 计划只覆盖部分内容：复用可确认部分，其余根据会话和代码补充。
- 找不到计划：根据当前会话需求和代码改动重新归纳。

文档中不要写“计划已验证”“本次已测试”等未实际执行过的结论。

## 输出位置与命名

统一输出到项目根目录下的功能目录：

```text
./agent-works/{feature-slug}/development-trace.md
```

每个功能需求一个目录，目录内固定使用 `development-trace.md` 作为留痕文件：

- 文档标题：中文功能需求名称。
- `{feature-slug}`：功能需求英文名，使用小写 kebab-case，例如 `risk-level-map-export`。
- 如果目标 `development-trace.md` 已存在，优先更新已有文件；更新时保留仍然准确的历史内容，删除或改写过期内容。

必要时先创建 `agent-works` 及对应功能目录。

## 确认流程

写入或更新 `agent-works` 文档前，必须先向用户确认留痕方案。

确认内容必须包含：

- 本次纳入留痕的文件范围：列出业务文件，说明排除的构建产物、临时文件或无关文件。
- 功能需求拆分结果：每个功能的中文标题、英文目录名 `{feature-slug}`、对应文件和拆分理由。
- 开发计划复用情况：列出找到的相关计划或说明未找到；说明每个功能准备复用、修正或重新归纳的依据。
- 输出动作：标明是新建文档还是更新已有文档，以及目标路径。

确认方式：

1. 先展示“拟生成/更新的开发留痕方案”。
2. 明确等待用户确认；用户未确认前不得创建或修改 `agent-works` 文档。
3. 如果用户要求调整文件范围、功能拆分、标题、目录名或目标路径，先按反馈更新方案，再次确认。
4. 只有用户明确表示确认、同意、继续、可以生成等含义后，才执行写入。

如果用户在最初请求中已经明确指定文件范围、功能拆分和输出文件名，也仍需在写入前做一次最终确认。

## 文档模板

每个文档使用以下结构：

```markdown
# 功能需求

## 功能的详细需求

- 写清楚业务目标、使用场景、输入输出、边界条件和用户可见行为。
- 如果需求来自开发计划或会话补充，融合成当前最终口径。

## 功能开发的实际结果

- 写清楚当前代码实际完成了什么。
- 列出关键业务文件及各自承担的职责。
- 如有未完成、暂未覆盖或依赖外部条件的部分，明确说明。

## 功能开发的具体实施方案

- 按实现链路描述具体怎么做：前端入口、状态/表单/交互、接口调用、后端接口、服务逻辑、数据查询、异常处理、配置等。
- 只写与本功能有关的实现细节，避免泛泛描述。
- 如果复用了原开发计划，说明最终实现与计划是否一致，以及差异点。
```

可以根据需要追加“关联文件”“验证情况”“注意事项”等章节，但前三个章节必须保留且内容完整。

## 执行流程

1. 定位项目根目录，查看 `git status --short` 获取当前改动范围。
2. 结合会话上下文确认本次需要留痕的业务文件；排除构建产物、依赖目录、临时文件和无关文档。
3. 查找相关需求文档、开发计划和已有 `agent-works` 文档。
4. 阅读必要 diff 和代码上下文，按功能需求分组。
5. 形成拟生成/更新方案，并按“确认流程”向用户确认。
6. 用户确认后，为每个功能生成或更新 `agent-works/{feature-slug}/development-trace.md`。
7. 完成后汇报生成或更新的文档路径、覆盖的功能需求、主要证据来源。

## 输出约束

- 文档内容使用中文。
- 未经用户确认留痕方案，不得创建或修改 `agent-works` 文档。
- 不复制敏感配置值；涉及配置时只写文件位置、用途和环境差异。
- 不夸大结果，不把推测写成事实。
- 不把验证写成已完成，除非本次确实执行了对应命令或脚本。
- 不为了留痕修改业务代码。
