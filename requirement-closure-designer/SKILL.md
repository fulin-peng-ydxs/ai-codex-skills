# 需求闭环设计器

## 核心定位

本技能不是独立重做一遍需求分析，而是 `requirement-doc-generator` 的“闭环模式”包装层。

执行本技能时，先完整读取 `~/.codex/skills/requirement-doc-generator/SKILL.md`，继承其中与以下事项相关的分析规则：

- 结合项目现状调研需求
- 以“资深业务专家 + 顶级架构师级别开发者”的视角补全需求
- 区分阻塞确认、非阻塞确认和推断决策
- 明确标注未知信息，不把猜测写成事实
- 优先沿用项目已有模式，不轻易引入新抽象

但要强制切换到“需求闭环设计”阶段，而不是正式需求文档阶段。

继承规则只用于调研、判断和不确定项分类；凡是与本技能收窄规则冲突的流程、输出结构或写入动作，一律以本技能为准。

## 本技能相对 requirement-doc-generator 的收窄规则

- 默认不要在首次响应时创建、覆盖或更新任何文档文件；先在对话中输出闭环分析、阻塞确认项和非阻塞确认项。
- 当前不要生成 `requirement.md`。
- 当前不要进入正式需求文档写作，不要主动展开文档章节、输出目录、需求编号、验收标准、性能安全兼容性等终态文档内容。
- 当前不要确定或创建正式需求文档目录，不要分配 `R1`、`R2` 等需求编号，不要把闭环分析整理成可直接落库的 `requirement.md` 终稿。
- 目标是补齐“这个需求在系统里怎么真正跑通”，不是把用户原话换一种方式复述，也不是提前写成 PRD。
- 只有当接口、数据、上下游系统或配置机制本身会直接影响闭环是否成立时，才说明这些内容；否则不要为了凑完整单独展开技术章节。
- 当用户明确确认闭环分析没有问题、确认闭环内容可落地，或要求保存、生成、更新闭环结果时，必须将最终闭环内容保存为闭环文档；闭环文档不是正式 `requirement.md`，不得顺带生成正式需求文档。
- 只有用户明确要求生成、保存或更新正式需求文档、PRD 或 `requirement.md` 时，才切换回 `requirement-doc-generator`。
- 用户仅确认闭环分析、认可方向、要求保存闭环文档或要求继续细化闭环时，不得自动进入正式需求文档生成。

## 闭环文档生成规则

当用户确认闭环无问题或要求保存闭环结果时，按以下规则写入最终闭环文档：

1. 写入时机。
   - 阻塞确认项未解决前，不要写入最终闭环文档，除非用户明确要求按当前假设保存。
   - 用户已经确认闭环无问题且输出路径可确定时，直接保存，不再追加一次确认。
   - 保存后只报告文件路径、是否覆盖或新建、仍需用户关注的遗留项；不要继续生成正式 `requirement.md`。

2. 输出目录。
   - 输出目录必须相对目标项目根目录，而不是无关的空对话目录、技能管理目录或 `~/.codex/skills`。
   - 如果当前工作区明显不是目标项目，或没有可用于调研的业务代码但用户要求结合项目现状，先确认目标项目路径，不要直接写入当前目录。
   - 如果用户指定了目录，使用用户指定目录。
   - 如果用户未指定目录，根据功能名生成精确 `{feature-slug}`，输出到 `./agent-works/{feature-slug}/requirement-closure.md`。
   - 同一功能已有 `./agent-works/{feature-slug}/` 目录时优先复用，确保闭环文档、正式需求文档、开发计划、开发留痕和业务核查后续可以归档在同一目录。
   - 生成前检查目标 `requirement-closure.md` 是否已存在；若已存在且用户没有明确要求更新该文件，不要覆盖。能够判断不是同一功能时，改用更精确的 `{feature-slug}`；能够判断是同一功能但是否更新不明确时，先确认。
   - 细化 `{feature-slug}` 时优先使用稳定业务语义，例如业务对象、流程、动作、页面、场景或配置范围；避免只用日期、随机数、`new`、`v2` 这类弱语义后缀。

3. 文件命名。
   - 默认文件名固定为 `requirement-closure.md`。
   - 只有用户明确指定其他文件名时，才使用用户指定文件名。
   - 不要把闭环文档命名为 `requirement.md`，不要在闭环文档阶段生成 `prototypes/`。

4. 写入内容。
   - 以用户最终确认的闭环分析为主要输入，保留已确认事实、阻塞确认结论、非阻塞确认项、推断依据、推断决策和范围划分。
   - 如果用户要求按当前假设保存，要在文档中标注哪些内容仍为假设或遗留确认项。
   - 不要在保存时重新扩写成正式需求文档，不要新增需求编号、正式验收标准、完整性能安全兼容性章节或页面 HTML 原型。

## 闭环分析目标

输出结果必须能回答清楚：

- 谁来用
- 从哪里进入
- 如何一步步完成操作
- 系统如何反馈
- 结果如何保存、如何生效、如何回退
- 失败时如何处理
- 最后如何判断闭环成立

重点是补齐完整功能闭环，而不是只复述核心诉求。

## 页面闭环规则

涉及页面时，闭环阶段必须先确定页面入口、页面职责、关键操作链路、状态流转、数据变化、权限边界和异常反馈；不要求在此阶段展开完整页面布局和全部交互细节，但不能把影响业务闭环的页面信息留到正式需求文档阶段再补。

具体约束如下：

- 如果页面、弹窗、抽屉、配置入口会影响业务闭环成立，必须在本阶段讨论清楚。
- 本阶段必须明确：有哪些页面或入口、谁从哪里进入、每个页面承担什么职责、页面之间如何衔接、联动和回流、有哪些关键操作链路、会触发哪些状态变化、数据变化或数据流转、权限边界以及异常、空状态和失败反馈。
- 本阶段不要求展开成正式页面设计稿，不要过早细化布局、字段顺序、按钮位置、文案、表格列宽、组件样式、交互微文案和视觉稿。

## 对话输出结构

除非用户指定其他格式，否则按以下结构输出：

```markdown
# 需求闭环分析

## 1. 初步需求理解

## 2. 业务目标

## 3. 用户角色、权限与使用场景

## 4. 完整业务闭环
- 主链路
- 关键支链路
- 状态流转
- 失败处理与回退方式

## 5. 完整功能点清单
对每个功能点说明：
- 功能目标
- 对应页面或入口
- 关键操作
- 系统反馈
- 状态变化
- 权限或配置要求
- 异常处理

## 6. 页面闭环草案
仅输出对闭环成立有直接影响的页面信息，包括：
- 页面/弹窗/抽屉/配置入口
- 页面职责
- 入口与跳转关系
- 页面之间的衔接、联动与回流关系
- 关键操作链路
- 状态变化
- 数据变化或数据流转
- 权限边界
- 异常、空状态与失败反馈

## 7. 阻塞确认项

## 8. 非阻塞确认项

## 9. 范围划分
- 必须实现
- 建议实现（每项标注处理状态）
- 暂不实现

建议实现的处理状态只使用：`本次纳入`、`后续建议`、`已实现`、`待确认`、`不纳入`。闭环阶段只能基于当前事实标注，不能把未落地事项写成已实现。

## 10. 推断依据与推断决策

## 11. 后续建议
说明当前是否已经具备保存闭环文档的条件，以及是否已经具备进入正式 `requirement.md` 生成的条件。
```

## 质量要求

- 优先补全业务闭环和系统落地路径，不要写成泛泛的 PRD 套话。
- 不要为了追求结构完整，提前固化未确认内容。
- 如果关键前提不明确，先列为阻塞确认项，不要强行编造。
- “完整功能点清单”和“页面闭环草案”不要重复抄写同一批内容；前者按功能链路组织，后者只补页面职责、页面衔接和页面层面的闭环信息。
- “建议实现”必须带处理状态，用于后续判断是否纳入、已做或暂不处理；状态缺失视为闭环信息不完整。
