# 前端复用优先清单

用于前端开发和代码审查。目标是防止页面私有实现、重复组件、重复状态逻辑和重复样式持续增长。

## 1. 查找入口

开发前先查：

- 同目录页面和组件。
- 全局组件目录、业务组件目录、布局组件目录。
- hooks/composables、utils、api helpers、constants、schemas、formatters。
- 全局样式、主题 token、设计系统、组件库、页面模板。
- 测试 helpers、fixtures、mock builders。
- `DESIGN.md`、页面规范、组件清单和项目级 AI 指令。

搜索线索：

- 页面标题、按钮文案、弹窗标题、空态文案。
- 字段名、接口参数、状态码、枚举值、标签文案。
- class 名、CSS 变量、颜色值、尺寸值。
- 表格列名、筛选项名、表单字段名。
- 函数名、事件名、store 字段、hook/composable 名。

## 2. 第二次出现判定

以下情况视为第二次出现，需要复用判断：

- 两个页面有相似筛选栏、工具栏、表格外壳、分页、操作列。
- 两个弹窗/抽屉有相似标题区、表单结构、footer、提交/取消逻辑。
- 两个页面有相似空态、加载态、错误态、权限态、提交中态。
- 两处代码有相同字段格式化、单位展示、枚举文案、状态颜色、tag/badge 映射。
- 两处接口调用有相同参数归一化、分页参数、排序参数、错误处理。
- 两处测试重复构造相同用户、列表数据、表单数据或页面状态。
- 两处样式重复表达同一种布局、间距、颜色语义或响应式策略。

判断重点是“同一变化原因”，不是代码是否完全一样。

## 3. 抽取类型选择

按最低必要层级抽取：

- 只是展示结构重复：抽展示组件。
- 展示结构重复且需要局部自定义：抽组件 + props/slots。
- 行为逻辑重复但 UI 不同：抽 hook/composable。
- 纯函数逻辑重复：抽 util。
- API 参数、分页、排序、筛选重复：抽 API helper 或 query builder。
- 表格列、筛选项、表单字段重复：抽 schema/config/factory。
- 状态文案、颜色、图标重复：抽 status map 或状态组件。
- 样式组合重复：抽 class/token/mixin。
- 测试准备重复：抽 fixture/test helper。

## 4. 抽象质量要求

好的复用单元：

- 名称描述稳定语义，而不是描述当前页面。
- 默认值覆盖主流场景。
- 差异点有明确 props、slots、options、callbacks 或 schema。
- 调用方代码比复制实现更短、更清楚。
- 两个以上消费点都能保持原行为。
- 测试覆盖共同逻辑和至少一个差异点。

坏的复用单元：

- 需要大量布尔参数才能工作。
- 把多个不相关职责塞进一个组件或 hook。
- 为了复用泄漏调用方内部状态。
- 让调用方传入整块临时 JSX/template 才能拼出页面。
- 命名绑定某个页面，却被多个模块使用。
- 抽象后每次新增差异都要改核心实现。

## 5. 复用实施步骤

1. 先锁定已有实现和新增需求的共同变化点。
2. 选择抽象层级。
3. 创建复用单元，保持 API 小而清楚。
4. 迁移旧使用点。
5. 接入新使用点。
6. 删除重复代码和页面私有样式。
7. 补测试或更新现有测试。
8. 若项目要求，更新 DESIGN、组件清单、文档或组件预览。
9. 运行项目要求的验证命令。

## 6. 不抽取说明

不抽取时说明：

- 重复点是什么。
- 已检查的同类实现在哪里。
- 为什么不是同一变化原因。
- 为什么抽取会增加复杂度或破坏分层。
- 后续第几次出现或哪个条件满足时再抽取。

示例：

```md
未抽取：两个页面都有“导出”按钮，但一个是同步下载，一个是异步任务创建并轮询进度，生命周期不同；当前只复用按钮样式，任务逻辑不合并。若后续出现第二个异步导出流程，再抽 `useAsyncExportTask`。
```

## 7. 审查定级

- `P0`：重复实现导致行为不一致、页面不可用、关键状态错误或测试失败。
- `P1`：已有明确复用组件/工具却新增页面私有实现；第二次出现但没有复用判断；重复逻辑会造成维护分叉。
- `P2`：重复样式、字段映射、状态文案、表格列、筛选配置或测试 helper，短期不阻断但会扩散。
- `P3`：可后续收敛的小重复或命名/抽象边界优化。

## 8. 和其他技能的边界

- 页面体验是否合理：交给页面 UX 审查。
- 是否遵循 DESIGN 文档：交给 DESIGN 遵循审查。
- 本清单只判断复用、重复、抽象边界和维护分叉风险。
