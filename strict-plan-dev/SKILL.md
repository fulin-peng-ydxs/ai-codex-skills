# Strict Plan Dev

## Overview

这个技能用于“已有需求文档和开发计划，但执行过程需要先对齐、再确认、再开发、再回填”的场景。

目标不是直接进入编码，而是按以下四步推进：

1. 锁定唯一需求文档与开发计划，并校验开发计划是否符合固定模板。
2. 根据固定计划章节整理本次待开发任务，给用户确认。
3. 按确认后的任务开发与测试，过程中持续遵守计划中的统一编码规范和模块级约束。
4. 按固定章节回填开发计划中的任务状态、完成情况、代码落点、验证结果和偏差信息。

## Trigger Guidance

以下表达应优先触发本技能或与本技能高度相关：

- “严格按开发计划执行”
- “先看需求和计划，再开始改”
- “先梳理当前开发情况和疑问”
- “先列开发任务给我确认”
- “边开发边更新开发计划”
- “做完后把计划状态回填”

若用户意图是“已有需求和计划，希望按计划推进，但不希望一上来直接编码”，优先使用本技能。

若用户只是让你直接修一个小问题，且没有要求对齐需求文档或开发计划，则不必强行使用本技能。

## Baseline Lock

开始执行时，先锁定唯一基线：

- 需求文档：优先查找 `docs/requirements/`
- 开发计划：优先查找 `docs/plans/`

若存在多个候选文档：

1. 先结合当前需求描述、文件名、最近改动和代码落点做一次匹配判断。
2. 如果仍无法可靠确定唯一基线，再向用户确认。
3. 不要混用多个需求或多个计划并行实施。

## Plan Template Alignment

在进入任务拆解前，必须先检查开发计划是否符合当前固定模板。

模板定义以 `generate-requirements-and-plan` 为唯一准绳：

- 需求与计划固定模板的唯一来源：`/Users/pengshuaifeng/.codex/skills/generate-requirements-and-plan/SKILL.md`
- `strict-plan-dev` 不重复定义整套模板正文，只负责校验、执行、回填。
- 若后续模板有调整，应以该技能中的最新定义为准，而不是以本技能中的历史描述为准。

进入任务拆解前，至少要校验以下维度：

- 需求文档的一级标题、顺序、章节分层是否符合当前固定模板。
- 开发计划的一级标题、顺序、章节分层是否符合当前固定模板。
- 开发计划中的后端、前端固定模块是否齐全，模块内部固定小节是否齐全。
- 需求文档中的强固定章节、半固定章节是否符合当前固定模板。
- 开发计划中的半固定章节是否回答了要求的最小问题集。

若当前开发计划不符合上述结构：

1. 不要直接跳过。
2. 先基于现有内容重组计划结构。
3. 在输出中明确说明重组了哪些章节。
4. 重组完成后再进入任务拆解。

若缺少半固定内容、模块内固定小节、任务进度记录结构或其他固定模板要求，也应先补齐或明确标记不完整，再进入任务拆解。

## Step 1: Understand Baseline

必须先完成以下动作，再进入任务拆解：

1. 阅读需求文档，提炼功能目标、范围、限制条件、验收点。
2. 阅读开发计划，提炼任务分解、阶段顺序、已记录状态、未完成项。
3. 校验开发计划是否符合固定模板；若不符合，先记录待修正点。
4. 校验需求文档是否符合固定模板；若不符合，先记录待修正点。
5. 检查实际代码，判断当前开发处于什么状态：
   - 未开始
   - 部分实现
   - 已实现但未验证
   - 已偏离计划
6. 输出一份简明理解结果，至少包含：
   - 对应需求文档与开发计划路径
   - 需求摘要
   - 计划摘要
   - 当前开发情况
   - 疑问、冲突点、缺失信息
   - 计划模板是否对齐
   - 需求文档模板是否对齐
   - 若不对齐，具体缺少哪些结构或最小问题集
7. 在进入任务拆解前，必须完成现有实现复用检查，至少识别：
   - 可复用的类、方法、组件、工具类、基础封装
   - 现有接口响应格式、异常处理格式、返回包装方式
   - 同类型功能的整体实现逻辑和代码组织方式
   - 本次应沿用、扩展还是重构的现有实现锚点
8. 若开发计划中存在 `统一编码规范`，必须先提炼本次开发准备遵守的规范清单，至少包括：
   - 项目风格统一要求
   - 参数与接口风格要求
   - 工具类、公共能力与现有实现复用要求
   - 注释要求
   - 明确的不允许事项
9. 若开发计划的具体模块章节中已写 `现有复用锚点`、`一致性要求`、`注释与可维护性要求`，也必须一并提炼，不能只看总则。
10. 必须区分：
   - 模块章节里的 `当前状态`：单个模块的当前判断
   - `任务进度记录`：整份计划的执行日志
   不得将两者视为同一字段。

## Step 2: List Tasks For Confirmation

在真正开发前，必须先给出“待确认的开发任务列表”。

任务列表必须按开发计划当前固定模块组织，而不是自由列任务。

- 后端任务：按开发计划中的后端固定模块顺序列出
- 前端任务：按开发计划中的前端固定模块顺序列出
- 若当前模板后续调整，应直接沿用计划文档中的最新固定模块顺序，不在本技能中重复维护一份清单

每个待确认任务必须说明：

- 任务目标
- 对应需求条目
- 对应计划章节
- 预计改动范围
- 复用锚点
- 一致性要求
- 注释与可维护性要求
- 适用的统一编码规范

若任务受需求文档中的验收、待确认或其他约束影响，还应显式标注对应约束条目。

如果发现原计划存在以下问题，应先更新计划文档，再输出任务列表：

- 一级标题不符合固定模板
- 后端或前端固定章节缺失
- 任务粒度过粗，无法直接执行
- 顺序不合理
- 缺少当前必须执行的步骤
- 状态与实际代码不一致
- 缺少复用方案或缺少一致性约束
- 缺少统一编码规范或规范内容过于空泛
- 模块级章节未写复用锚点、一致性要求、注释要求
- 需求文档未按当前固定模板组织

此步骤的输出应明确区分：

- 已在计划中的任务
- 本次新增或修正到计划中的任务
- 需要用户确认后才能继续的任务

用户未确认前，不进入大规模开发。

## Step 3: Develop And Test

在用户确认任务后，再开始实施。

执行要求：

1. 只开发已确认任务，避免顺手扩展需求。
2. 尽量按计划顺序实施；若实际顺序调整更合理，应同步更新计划。
3. 每完成一项任务，做对应验证，不积压问题。
4. 开发前必须先搜索项目内可复用实现，优先复用已有工具类、公共方法、公共组件、统一封装和相似业务实现。
5. 接口响应格式、错误处理方式、字段组织方式、整体实现逻辑应与项目现有同类模块保持一致。
6. 若未复用现有实现或未沿用现有风格，必须在输出和计划回填中明确说明原因。
7. 若开发计划中已定义 `统一编码规范`，开发时必须逐项遵守；如确需偏离，必须先记录原因并同步更新计划。
8. 若当前任务对应某个具体计划模块，则必须同时遵守该模块下的：
   - 现有复用锚点
   - 一致性要求
   - 注释与可维护性要求
9. 若当前任务受需求文档中的验收、待确认或其他约束影响，开发与验证后必须能够用本轮结果说明这些约束已覆盖、未覆盖或仍受阻塞。
10. 若当前任务仍受未确认信息影响，未确认前只能按当前默认处理推进，并在输出与回填中明确标注。
11. 若计划中写明某模块“无改动”，则不得在未更新计划的情况下擅自改动该模块。
12. 若某模块状态发生变化，既要更新模块章节中的 `当前状态`，也要在 `任务进度记录` 中追加本轮记录。

测试要求按改动范围取最小充分集：

- 后端改动：至少完成编译、相关单测或局部验证、接口或关键逻辑验证
- 前端改动：至少完成相关静态检查、局部构建或页面/交互验证
- 前后端联动：校验接口字段、类型、错误处理、空值语义、关键异常路径

开发阶段以“完成与当前任务匹配的最小充分验证”为准；如需执行统一质量审查、风格一致性检查和优化，交由 `code-review-improve` 执行。

## Step 4: Update Plan Status

开发与测试完成后，必须回填开发计划。

回填时不得只更新“已完成/未完成”状态，必须回填到对应固定章节中。
不得新建新的一级标题或平级替代章节，例如：

- 不得新增 `本轮开发结果`
- 不得新增 `本次回填`
- 不得新增 `开发完成情况`
- 不得新增任何用于替代现有固定章节的平级章节

回填位置必须严格限定为以下位置：

- 模块状态回填：只能回填到开发计划中对应的后端或前端开发模块既有小节中，重点更新该模块下的 `当前状态` 以及必要说明内容。
- 执行轨迹回填：只能回填到开发计划中的 `任务进度记录` 章节及其固定子结构。
- 若需补充本轮的验证、阻塞、偏差、备注，也必须写入 `任务进度记录` 的本轮记录位置，不得单开新章节。

至少更新以下内容：

1. 任务状态
   - 未开始
   - 进行中
   - 已完成
   - 已阻塞
2. 完成情况
   - 做了什么
   - 实际代码落点
   - 实际验证结果
3. 模块级执行情况
   - 实际复用了哪些锚点
   - 实际遵守了哪些一致性要求
   - 注释与可维护性要求是否落实
   - 若未复用或有偏差，原因是什么
4. 统一编码规范执行情况
   - 已遵守的规范
   - 存在偏差的规范
   - 偏差原因
5. 若某任务未完成，要写清原因和后续建议。

若实际实施与原计划有偏差，需要在计划中保留这种偏差的说明，而不是只改状态不解释原因。

回填时还必须满足：

- 模块章节中的 `当前状态`：反映该模块在本轮结束后的最新状态。
- `任务状态总览`：汇总所有后端和前端固定章节的最新状态。
- `本轮更新记录`：按日期或轮次追加一次执行记录，记录本轮完成项、进行中项、阻塞项、代码落点和验证结果。
- 若某模块本轮无实际变更，则只更新必要状态，不得为了凑格式新增无意义章节。

两者关系如下：

- 模块 `当前状态` 是当前快照。
- `任务进度记录` 是历史流水。
- 只更新其中一部分，视为回填不完整。

## Working Rules

1. 先理解和对齐，再编码。
2. 先列任务给用户确认，再进入实现。
3. 计划可以更新，但更新必须基于实际情况，并在输出中明确说明。
4. 保持最小必要改动，避免无关重构。
5. 所有结论都尽量落到文档、代码、测试结果，不凭空假设。
6. 输出尽量使用固定模板，减少自由发挥。
7. 新增代码前必须先检索项目内同类实现和可复用封装，优先复用而不是重复造轮子。
8. 工具类、公共方法、响应包装、异常处理、DTO/VO 组织、页面交互模式等应尽量沿用现有项目风格，保持整体一致。
9. 如确实无法复用现有实现，必须记录“不复用原因”和“新实现与现有风格的对齐方式”。
10. 若开发计划中存在 `统一编码规范`，则该章节属于开发执行基线，不能在实现阶段被忽略。
11. 若开发计划中存在具体模块章节约束，则这些约束与统一编码规范同样属于执行基线，不能在实现阶段被忽略。
12. 模块 `当前状态` 与 `任务进度记录` 必须同步维护；前者是状态快照，后者是历史记录，不能互相替代。

## Code Comment Rules

生成代码时，注释要求采用高说明度标准。以下内容默认都要有明确、详细、可读的注释交代：

1. 每个类：说明职责、使用场景、与上下游模块或对象的关系。
2. 每个字段：说明字段含义、业务语义、取值约束、特殊约定或单位。
3. 每个方法或函数：说明用途、核心参数、返回值、前置条件、关键副作用或异常语义。
4. 核心业务逻辑代码块：说明业务意图、执行流程、关键判断依据、边界处理、兼容原因。

注释规则：

1. 注释要优先解释业务背景、设计意图和约束条件，不能只把代码逐句翻译一遍。
2. 方法或函数注释不能只写“查询数据”“保存信息”这类空话，要写清楚处理对象、处理条件和结果语义。
3. 字段注释不能只写字段字面意思，要补充业务用途、状态含义、枚举取值、来源或去向。
4. 核心业务逻辑前要有成段说明，必要时在关键分支处继续补充行内或邻近注释。
5. 兼容旧逻辑、临时兜底、协议映射、历史数据修正、异常绕行等代码，必须明确写出原因，避免后续误删。
6. 注释应紧贴声明或逻辑本身，保持结构化和可维护，不允许用无信息量的模板化注释凑数。
7. 在 Step 3 开发完成后，应自查本次新增或修改的类、字段、方法、核心业务逻辑是否都已补齐详细注释。

## Interaction Contract

执行时默认遵循下面的交互约束：

1. 完成 Step 1 后，必须先汇报“基线理解结果”，不能直接开始开发。
2. 完成 Step 2 后，必须先给出“待确认开发任务”，等待用户确认。
3. 只有在用户确认任务后，才能进入 Step 3 的实际开发。
4. Step 4 的计划回填应与实际开发结果一致，不能只更新状态不写完成情况。

## Output Format

按以下顺序组织输出：

1. 基线理解结果
   - 需求文档
   - 开发计划
   - 当前开发情况
   - 计划模板对齐情况
   - 疑问和风险
2. 待确认开发任务
   - 按计划章节列出的任务清单
   - 对计划的新增或修正点
3. 开发与测试结果
   - 已完成任务
   - 核心改动
   - 验证结果
4. 计划回填结果
   - 更新后的任务状态
   - 模块级回填情况
   - 未完成项与原因
