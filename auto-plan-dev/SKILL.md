# 开发计划生成器

## 强制调用规则

只有当用户明确提出使用 `auto-plan-dev`、`$auto-plan-dev` 或“使用开发计划生成器技能”时，才允许使用本技能。用户只是要求生成开发计划、实施计划、任务拆解或 `plan.md` 时，不得自动启用本技能。

## 核心目标

把已确认的需求文档或功能需求转成一份可直接指导 AI 或开发者实施的 `plan.md`。计划必须体现 AI 计划模式的风格：先理解现状和约束，再拆解有序任务，明确验证标准、风险、依赖和进度追踪方式。

## 执行流程

1. 确定输入和输出目录。
   - 优先读取用户指定的需求文档，例如 `requirement.md`、PRD、需求分析文档或用户提供的功能说明。
   - 输出目录必须相对目标项目根目录，而不是无关的空对话目录或技能管理目录。
   - 如果当前工作区明显不是目标项目，或缺少可用于调研的业务代码但用户要求结合项目现状，先确认目标项目路径。
   - 如果用户指定了目录，使用用户指定目录。
   - 如果用户未指定目录，根据功能名生成 slug，输出到 `./agent-works/{feature-slug}/plan.md`。
   - 同一功能已有 `./agent-works/{feature-slug}/requirement.md`、开发留痕、业务核查、SQL 或测试记录时，必须复用该 `{feature-slug}` 目录。

2. 读取需求并补足项目上下文。
   - 先完整阅读需求文档，提取目标、范围、角色、流程、数据、接口、权限、验收标准和遗留确认项。
   - 按可追踪的需求项拆分需求，每个需求项必须保留来源、核心内容、验收意图和是否属于本次范围。
   - 调研相关代码、页面、接口、服务、权限、配置、数据库脚本和测试。
   - 如果需求涉及页面、布局、组件、交互、弹窗/抽屉、表格、图表、表单或配置入口，必须检查项目是否已有设计规范、组件库、页面模板、布局约定、样板页面、视觉验收方式或前端测试契约；有则纳入计划依据，没有则在计划中声明本次采用的最低 UI 约束。
   - 如果需求文档包含页面与交互开发需求、页面布局图、功能图、变更区域图或线框图，计划必须承接其中的页面、页面功能点、区域、模块、操作路径和状态；如果需求涉及页面但缺少图示或页面功能点清单，应将其列为阻塞或待补需求，不要只凭文字拆任务。
   - 如当前环境存在数据库 MCP 或可用数据库信息，结合表结构和实际数据判断计划中的数据改动。
   - 在计划中记录关键依据，例如文件路径、类名、方法名、接口名、表名、字段名、枚举值和已有业务规则。
   - 如果无法访问数据库或关键上下文，要明确写入限制和假设，不要编造事实。

3. 判断是否存在阻塞问题。
   - 阻塞问题包括会改变业务流程、数据结构、权限范围、状态流转、接口契约、上线步骤或验收标准的不确定项。
   - 存在阻塞问题时，先向用户确认，不要直接生成最终计划。
   - 非阻塞问题可写入“假设与待确认项”，但必须说明对实施的影响和后续确认时机。

4. 拆解实施路径。
   - 按依赖顺序组织任务，优先识别基础模型、数据结构、接口契约、后端逻辑、前端交互、权限、兼容、测试和发布步骤之间的先后关系。
   - 每个任务必须足够具体，能让开发者知道改哪里、做什么、如何验证。
   - 建立需求项与开发任务的映射关系，确保每个“必须实现”的需求项都有对应任务、产出和验证方式；任何没有需求依据但为了实现所必需的技术任务都要说明理由。
   - 涉及页面开发时，必须把需求文档中的页面功能点逐项映射到技术实现，说明组件、状态、事件、数据/API、校验、权限、异常和联动处理。
   - 避免把计划写成泛泛的“实现功能、联调测试、上线发布”。每项任务都要有输入、产出、依赖和验证方式。
   - 涉及新页面或复杂页面时，必须先拆出页面结构决策：页面类型、主操作区、主内容区、辅助区、弹层/抽屉、固定区域、滚动区域、主要增长区域、小屏或低高度退化方式。
   - 涉及配置化、可定制、偏好记忆、模板或预设能力时，必须拆出配置闭环任务：配置模型、默认值、权限或作用域、校验、存储、读取、消费、重置、兼容、测试。只保存但没有消费路径的配置项视为计划不完整。
   - 当需求会被多个页面、模块或业务场景复用时，必须按基础能力建设计划：先做核心模型和约束，再做至少一个真实样板接入，最后规划渐进迁移或接入边界，避免一次性铺开。

5. 写入最终开发计划。
   - 必要时创建输出目录。
   - 默认文件名为 `plan.md`。
   - 如果计划涉及数据库变更、初始化数据、字典、菜单、权限、配置或迁移 SQL，必须明确 SQL 文件路径、用途、执行时机、影响范围、回滚方式或不可回滚原因；默认路径为 `./agent-works/{feature-slug}/{sql-purpose}.sql`。
   - 如果用户确认按该计划执行开发，开发完成时可将 AI 自身计划归档到 `~/.codex/plans/{plan-slug}.md`；这只用于 AI 自身计划留存，不替代项目内 `./agent-works/{feature-slug}/plan.md`。

## 需求映射要求

- 需求编号必须稳定、可追踪，可使用 `R1`、`R2` 这类本计划内编号；同一需求项在映射表、任务表、详细步骤和验证计划中必须使用同一编号或任务编号。
- 每个“必须实现”的需求项至少对应一个任务；同一个任务可以覆盖多个强相关需求项，但必须在映射表中逐项列出。
- 每个任务必须能回溯到至少一个需求项、现有系统约束、验收标准或必要技术依赖；不能回溯的任务必须标为 `超范围` 并说明取舍。
- 映射表中的 `部分覆盖`、`未覆盖`、`超范围`、`待确认` 不得只停留在说明里，必须反映到任务、风险、假设、待确认项或阻塞处理。
- 更新计划状态时，同步更新映射表和任务编排表；不能只更新任务表导致需求覆盖状态失真。

## 任务编排要求

- 任务粒度要能直接用于实时追踪，避免一个任务覆盖多个无依赖关系的大块工作。
- 每项任务必须有明确产出，例如“新增接口”“调整页面字段”“补充权限校验”“新增测试用例”“编写 SQL 文件”。
- 任务顺序必须体现依赖关系，不能把前端联调放在接口契约确定之前，不能把回归测试放在实现任务之前。
- 状态初始值默认全部为 `待开始`；如果计划生成时已经完成部分调研或文档产出，可标为 `已完成`，但要基于真实已完成工作。
- 遇到未确认的非阻塞事项，给对应任务标注依赖或风险；遇到阻塞事项，不生成最终计划。
- 执行过程中更新计划时，只更新真实状态，不要批量把所有任务一次性标为完成。
- 任务编号必须与“需求与开发内容映射和开发状态”章节中的对应任务保持一致。

## 质量要求

- 优先沿用当前项目已有模式，不轻易引入新框架、新依赖或新抽象。
- 计划必须能被执行，而不是只描述方向。
- 计划必须能说明每个开发内容从哪里来、服务哪个需求或约束；不能把未确认的想法包装成需求内任务。
- 涉及前端时覆盖页面路径、组件状态、空状态、加载状态、错误状态、权限状态和兼容表现。
- 涉及前端页面时，第 4 章技术方案必须逐项回应需求文档中的页面功能点，说明具体技术实现；不能只写“按需求实现页面”。
- 涉及新页面或复杂页面时覆盖页面结构、主次区域、滚动/固定策略、弹层入口、响应式或低高度退化，并承接需求文档中的布局图和功能图。
- 涉及后端时覆盖接口、服务、事务、权限、日志、异常、幂等、数据一致性和性能影响。
- 涉及数据库时覆盖字段、索引、数据迁移、初始化数据、回滚和历史数据兼容。
- 涉及第三方系统时覆盖契约、失败重试、超时、降级、审计和联调方式。
- 涉及配置化、可定制、偏好记忆、模板或预设时，必须覆盖“谁定义、谁渲染、谁保存、谁读取、谁消费、谁校验、如何回退”，并把相关任务落到具体文件、模块或接口。
- 涉及基础能力、框架化能力或跨页面复用能力时，必须包含真实样板场景验证；只有抽象代码、没有真实接入样板的计划不完整。
- 测试计划必须可观察、可执行、可复现。
- 明确标注未知信息和推断依据，不要把猜测写成事实。
