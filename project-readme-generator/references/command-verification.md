# 命令验证

## 门禁原则

README 中作为常规流程推荐的命令必须在本次任务中实际执行成功后才能写入。非零退出、缺少工具、健康检查失败、端口未就绪或超时都视为失败。

门禁失败时，不覆盖 `README.md`。可以在对话中汇报问题，或在用户明确同意后写独立草案文件。

## 验证计划

编辑 README 前，先列出要验证的命令：

- `source`：命令来源，如清单文件、启动脚本、已有 README、CI 配置、Makefile。
- `cwd`：工作目录。
- `command`：精确命令。
- `purpose`：环境、依赖、测试、构建、启动、健康检查、停止、CLI、数据准备。
- `required_for_readme`：是否会作为 README 常规流程写入。
- `success_condition`：退出码、预期输出、端口、HTTP 状态、生成产物或测试结果。
- `cleanup`：长期服务的停止和状态恢复方式。

破坏性、生产环境、写外部服务、耗时昂贵、需要密钥或会改写数据的命令，先询问用户。

## 必验范围

至少验证：

- 运行时版本和包管理器版本。
- README 会写入的安装或依赖检查命令。
- README 会写入的测试命令。
- README 会写入的构建命令。
- README 会写入的本地启动、状态、健康检查和停止命令。
- README 会写入的关键 CLI 或数据准备命令；若命令会写库、联网或耗时，先询问用户。

## 长期服务

验证启动命令时：

1. 先检查是否已有服务运行。
2. 优先使用项目封装脚本。
3. 只启动到足够验证就绪。
4. 用状态命令、端口、HTTP 响应或健康检查确认。
5. 只停止本次验证启动的进程。
6. 项目要求恢复测试前状态时，按测试前状态恢复。

## 失败处理

失败后：

1. 记录命令、工作目录、退出状态和最短有效错误。
2. 分类为缺依赖、版本不符、测试失败、构建失败、端口冲突、外部服务缺失、密钥缺失、数据锁、超时或未知。
3. 只有存在明确、非破坏性修复依据时重试一次。
4. 重试失败或修复方式不确定时停止并询问用户。
5. 不连续试命令变体。

README 不应把失败命令写成已验证流程。必要时可写“需要先完成某前置条件”，但不要假装命令可直接运行。
