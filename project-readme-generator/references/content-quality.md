# 内容质量检查

## 写作原则

- 面向真实读者写，不面向作者自己的记忆写。
- 第一屏说明项目是什么、解决什么问题、谁会用、最终得到什么。
- 命令、版本、端口、入口和目录必须来自当前仓库事实。
- 默认使用简体中文；命令、代码、文件名、机器字段按原文。
- 用表格组织模块、命令、工作流和文档入口。
- 用 Mermaid 表达架构、数据流或任务流；只有关系复杂时才加图。
- 保持 README 是入口，不把架构、设计、API、需求、变更记录全部塞进来。

## 去冗余

删除或外移：

- 历史迁移过程、阶段流水、旧计划、临时背景。
- 与代码不一致的旧命令、旧端口、旧架构。
- 大段复制的下游技术文档。
- 对 AI 或开发者无帮助的口号。
- 本地私密路径、密钥、私人账号和不可复用环境细节。

保留：

- 当前项目定位、能力范围、运行入口。
- 会影响使用和开发的环境、命令、配置、服务依赖。
- 会破坏正确性、安全、数据、权限、兼容性或构建产物的硬约束。
- 指向权威文档的链接。

## 链接与格式

- 相对链接必须指向仓库内真实文件或目录。
- 外链只在必要时保留。
- Mermaid 图中的节点文本要简洁。
- 表格列数不要过多；超过 5 列时考虑拆分。
- 代码块必须标注语言，如 `bash`、`mermaid`、`json`。
- 命令块里不要混入不可执行的长解释；解释放在命令前后。

## 自检清单

落地前确认：

- 读者能在 5 分钟内理解项目价值和运行路径。
- 所有 README 常规命令已验证或已明确说明前置条件。
- 技术栈、目录、端口和文档入口与仓库当前事实一致。
- 没有复制来源项目业务内容。
- 没有把 AI 协作规则、详细设计或历史记录当作 README 主体。
