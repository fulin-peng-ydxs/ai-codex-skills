# README 模型

## 核心定位

README 是项目读者的第一入口，不是所有文档的合集。它应该让读者快速判断项目价值、运行方式、主要能力、技术结构、验证路径和深入阅读入口。

不要固定复刻某个项目的章节。按目标项目事实选择模块，模块标题可根据项目语境调整。

## 当前项目可借鉴的通用写法

从复杂全栈项目 README 中可抽象出这些稳定方法：

- 开头先说明项目定位、目标读者和解决的问题。
- 用表格把“问题/场景 -> 系统能力 -> 产出”说清楚。
- 写一条完整使用闭环，让读者理解功能之间如何串起来。
- 用少量设计主线解释系统为什么这样组织。
- 架构段落说明前端、后端、持久层、外部服务和数据流。
- 模块表同时给页面入口、API/CLI 入口、用途和详细文档链接。
- 常用工作流按场景组织，而不是只堆命令。
- 目录结构只列影响开发定位的路径，不把所有文件树展开。
- 技术栈和版本要求来自清单文件和实际验证。
- 服务命令分开发、生产、测试、构建、数据准备、故障排查。
- 开发契约只写不可破坏的不变量，细节指向专业文档。
- 最后给当前文档入口，避免 README 承载所有细节。

这些是表达方式，不是业务内容；不要复制来源项目的领域规则、模块名、端口、账号或数据口径。

## 基础模块

大多数项目可从这些模块中选择：

1. `项目标题与一句话定位`
   - 写项目名、核心用途、目标用户或运行场景。
   - 标题不要堆满技术名词；技术栈可放到后文。

2. `解决的问题 / 适用场景`
   - 用短段落或表格说明项目解决什么问题、典型使用者是谁、最终产出是什么。
   - 工具库可写能力范围；业务系统可写使用闭环；服务端项目可写 API/任务边界。

3. `快速开始`
   - 给最短可运行路径。
   - 命令必须通过验证；若依赖外部服务或密钥，先写前置条件。

4. `功能模块 / 能力清单`
   - 对应用系统：列页面、接口、CLI、任务、数据流和详细文档入口。
   - 对库：列核心 API、示例、扩展点和兼容版本。
   - 对 CLI：列常用命令、输入输出和典型工作流。

5. `架构与数据流`
   - 只在项目结构复杂或多组件协同时写。
   - 可用 Mermaid 表达组件关系、请求路径、任务流或数据流。

6. `目录结构`
   - 只列读者需要定位代码、配置、测试、文档、构建产物的目录。
   - 说明职责边界，不展开无关生成物。

7. `技术栈与版本`
   - 来自 `package.json`、`pyproject.toml`、`requirements.txt`、`pom.xml`、`build.gradle`、Docker 或配置文件。
   - 区分要求版本和本机验证版本。

8. `运行、测试与构建`
   - 安装、开发启动、测试、构建、预览、部署按类别分组。
   - 长期服务要写端口、健康检查、日志、停止方式。

9. `配置与环境变量`
   - 写必要配置、示例文件、敏感信息处理方式。
   - 不写真实密钥、私有账号或用户本机密码。

10. `常见问题 / 故障排查`
    - 只写当前项目常见且可操作的问题。
    - 优先给检查顺序、日志位置、状态命令和恢复方式。

11. `开发契约 / 贡献说明`
    - 只写会影响正确性、安全、兼容、数据、权限、生成产物的硬约束。
    - 普通代码风格细节可指向专门文档或配置文件。

12. `文档入口`
    - 链接架构、设计、API、需求、AI 协作、部署等权威文档。
    - README 只做导航和摘要。

## 条件模块

仅在目标项目确实需要时加入：

- `截图或演示`：UI 产品、组件库、可视化工具、游戏或设计导向项目。
- `API 示例`：SDK、服务端 API、网关、数据服务。
- `数据准备 / 迁移`：项目依赖本地数据库、数据文件、迁移脚本或采集任务。
- `权限与账号`：项目有登录、RBAC、多租户、固定测试账号或权限验收。
- `部署与运维`：项目有镜像、CI/CD、反向代理、TLS、定时任务或生产环境说明。
- `兼容性矩阵`：库或插件需要声明运行时、平台、框架版本兼容。
- `路线图`：用户明确要求或项目维护者确实维护路线图；不要写猜测计划。
- `许可证与引用`：开源项目、模板、数据集、论文或第三方资产需要说明来源和许可。

## 结构控制

- 小项目：标题、定位、快速开始、使用、测试构建、目录、许可证即可。
- 中型应用：增加模块、架构、配置、故障排查、文档入口。
- 复杂全栈/数据系统：增加工作流、数据流、服务生命周期、开发契约和外部服务。

README 应该能读完并行动。详细设计、长表、历史记录和内部过程应移到专门文档。
